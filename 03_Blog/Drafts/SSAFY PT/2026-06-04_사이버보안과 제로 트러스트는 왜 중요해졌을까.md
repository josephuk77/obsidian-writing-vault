# 사이버보안과 제로 트러스트는 왜 중요해졌을까

## 개념 한 줄 정리

| 개념 | 간단한 설명 |
| --- | --- |
| 사이버보안 | 시스템, 네트워크, 데이터, 계정을 공격으로부터 보호하는 기술과 절차 |
| 제로 트러스트 | 내부망도 기본적으로 신뢰하지 않고 모든 접근을 계속 검증하는 보안 모델 |
| 인증 | 사용자가 누구인지 확인하는 과정 |
| 인가 | 인증된 사용자가 어떤 권한을 갖는지 확인하는 과정 |
| MFA | 비밀번호 외에 추가 인증 수단을 요구하는 방식 |
| 최소 권한 | 필요한 권한만 부여해 피해 범위를 줄이는 원칙 |
| API 보안 | API 호출을 인증, 권한, 속도 제한, 로그로 보호하는 방식 |

## 글을 쓰는 이유

보안은 모든 IT 서비스의 기본이다.

특히 클라우드, 원격근무, IoT, AI, API가 늘면서 보호해야 할 대상이 많아졌다. 예전처럼 회사 내부망만 지키면 되는 구조가 아니다.

SSAFY PT 면접에서 보안을 주제로 잡으면 개발자 관점의 해결방안을 말하기 좋다. 단순히 "보안 의식을 높여야 한다"가 아니라 MFA, 최소 권한, API Gateway, Rate Limit, 감사 로그, 네트워크 분리 같은 구체적 기술을 제시할 수 있기 때문이다.

## 먼저 결론부터

제로 트러스트는 "아무도 기본적으로 믿지 않는다"는 보안 접근이다.

회사 내부망에 있는 사용자라도 계속 검증하고, 사용자 권한도 필요한 만큼만 부여한다. 계정이 탈취되거나 내부망에 침투가 발생해도 피해가 전체 시스템으로 퍼지지 않게 만드는 것이 목적이다.

## 개념

기존 보안은 경계 방어에 가까웠다. 내부망과 외부망을 나누고, 외부에서 들어오는 공격을 막는 데 집중했다.

하지만 지금은 사용자가 집, 카페, 회사, 모바일 환경에서 접속한다. 서비스는 클라우드에 있고, API는 외부 시스템과 연결된다. IoT 기기도 네트워크에 붙는다.

그래서 "내부망이면 안전하다"는 가정이 약해졌다.

제로 트러스트는 다음 원칙을 가진다.

- 모든 접근을 검증한다.
- 사용자, 기기, 위치, 권한을 함께 본다.
- 권한은 최소한으로 부여한다.
- 접근 기록을 남기고 이상 행위를 감지한다.
- 내부망 침투 이후 확산을 줄인다.

## 등장 배경

- 원격근무와 클라우드 사용이 늘면서 접속 위치가 다양해졌다.
- 계정 탈취와 피싱 공격이 증가했다.
- API와 SaaS 사용이 많아져 외부 연동 지점이 늘었다.
- IoT 기기와 모바일 기기가 네트워크에 많이 연결되었다.
- 내부망 침투 이후 피해가 확산되는 보안 사고가 많아졌다.

정리하면 제로 트러스트는 네트워크 경계가 흐려진 환경에서 등장한 보안 모델이다.

## 장점

- 계정 탈취 피해를 줄일 수 있다.
- 사용자, 기기, 위치, 권한을 계속 검증할 수 있다.
- 내부망 침투 이후 확산을 줄일 수 있다.
- 클라우드와 원격근무 환경에 잘 맞는다.
- API와 관리자 권한을 더 안전하게 보호할 수 있다.

## 단점/리스크

- 초기 구축 비용과 운영 복잡도가 크다.
- 인증 절차가 많아지면 사용자 경험이 나빠질 수 있다.
- 기존 시스템과 연동이 어려울 수 있다.
- 정책을 잘못 설정하면 정상 사용자가 차단될 수 있다.
- 로그와 정책이 많아져 관리 부담이 커질 수 있다.

## 기술적 해결방안

| 리스크 | 기술적 해결방안 |
| --- | --- |
| 계정 탈취 | MFA, 패스키, 이상 로그인 탐지, 세션 만료 |
| 과도한 권한 | RBAC/ABAC, 최소 권한, 권한 주기적 검토 |
| 내부망 확산 | 마이크로 세그멘테이션, 네트워크 분리, 접근 제어 |
| API 공격 | API Gateway, Rate Limit, JWT 검증, WAF |
| 보안 로그 누락 | SIEM, 감사 로그, 알림 규칙, 이상 행위 탐지 |
| 사용자 불편 | 위험 기반 인증, SSO, 조건부 접근 |
| 정책 오류 | 단계적 적용, 테스트 환경 검증, 롤백 정책 |

예를 들어 관리자 페이지는 일반 로그인만으로 접근하게 하면 위험하다. MFA를 적용하고, 관리자 권한을 최소화하고, 모든 관리자 행동을 감사 로그로 남겨야 한다.

API는 인증 토큰을 검증하고, 호출 횟수를 제한하고, 비정상 요청은 WAF나 API Gateway에서 차단할 수 있다.

## 활용사례

- 사내 시스템 로그인 보안
- 클라우드 관리자 권한 보호
- API 호출 제한과 인증
- IoT 기기 접근 제어
- 금융/공공 서비스 접근 제어
- 관리자 페이지 접근 통제

## 예상 질문과 답변

### 제로 트러스트란 무엇인가요?

내부망이라고 해서 무조건 신뢰하지 않고, 사용자와 기기, 위치, 권한을 계속 검증하는 보안 모델입니다.

### 개발자가 보안에서 할 수 있는 일은 무엇인가요?

인증과 인가를 분리하고, 최소 권한을 적용하고, API 요청을 검증하고, 중요한 행동은 감사 로그로 남길 수 있습니다. 관리자 기능에는 MFA와 접근 제한을 적용할 수 있습니다.

### 사용자 불편은 어떻게 줄일 수 있나요?

위험 기반 인증을 적용할 수 있습니다. 평소와 같은 위치와 기기에서는 간단히 로그인하고, 낯선 위치나 위험한 요청에서는 추가 인증을 요구하는 방식입니다.

## 정리

보안은 개발 이후에 붙이는 기능이 아니라 서비스 설계 단계에서 함께 고려해야 하는 요소다.

면접에서 사용할 결론:

> 제로 트러스트는 내부망도 무조건 신뢰하지 않고 모든 접근을 검증하는 보안 모델입니다. 클라우드, 원격근무, API, IoT가 늘어난 환경에서는 MFA, 최소 권한, API Gateway, 감사 로그, 네트워크 분리 같은 기술적 대응을 통해 보안 리스크를 줄여야 합니다.
