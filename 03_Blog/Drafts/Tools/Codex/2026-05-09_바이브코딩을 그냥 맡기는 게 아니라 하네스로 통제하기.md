# 바이브코딩을 그냥 맡기는 게 아니라 하네스로 통제하기

## 글을 쓰는 이유

요즘 AI 코딩 도구를 쓰면 예전보다 훨씬 빠르게 코드를 만들 수 있다.

대충 만들고 싶은 기능을 설명하면 파일도 만들고, 컴포넌트도 만들고, 테스트도 어느 정도 작성해준다. 이런 흐름을 흔히 바이브코딩이라고 부른다.

처음에는 이 방식이 굉장히 편하게 느껴졌다. 머릿속에 있는 아이디어를 말로 던지면 코드가 생기기 때문이다.

그런데 수업에서 하네스 구조를 보고 나서 생각이 조금 바뀌었다.

AI에게 코드를 맡기는 것 자체보다 중요한 것은, AI가 어떤 순서로 생각하고 어떤 기준으로 검증하게 만들 것인가였다.

이번 글은 내가 수업에서 배운 하네스 기반 바이브코딩 흐름을 정리한 글이다.

관련 정리 노트는 [[Codex TDD 하네스 전환 계획]]에 따로 모아두었다.

## 먼저 결론부터

바이브코딩은 “AI에게 알아서 만들어달라고 맡기는 방식”이 아니라 “작업 단위를 작게 나누고, 문서와 테스트와 훅으로 AI를 통제하는 방식”에 가까워야 한다.

내가 정리한 핵심은 이렇다.

- 코딩 전에 기획 문서를 먼저 만든다.
- 한 번에 전체 구현을 맡기지 않는다.
- phase와 step으로 작업을 작게 나눈다.
- 각 step은 독립된 세션에서도 실행 가능할 만큼 자세해야 한다.
- TDD를 기본값으로 둔다.
- lint, build, test를 커밋 전 반드시 실행하게 만든다.
- 화면이 있는 작업은 브라우저 테스트까지 한다.
- 문서에 적힌 규칙만 믿지 않고 hook으로 강제한다.

즉, 바이브코딩의 핵심은 “느낌대로 빠르게 만들기”가 아니라 “AI가 사고칠 수 있는 범위를 줄이면서 빠르게 만들기”라고 느꼈다.

## 바이브코딩에서 위험한 지점

AI 코딩 도구를 쓰면 속도는 빨라진다.

하지만 속도가 빨라지는 만큼 잘못된 코드도 빠르게 쌓일 수 있다.

예를 들어 이런 문제가 생길 수 있다.

- 처음 요구사항과 다른 방향으로 기능이 커진다.
- 테스트 없이 구현부터 한다.
- 파일 여러 개를 한 번에 수정해서 어디서 문제가 생겼는지 찾기 어렵다.
- 기존 아키텍처 규칙을 무시한다.
- 에러는 없어 보이지만 실제 브라우저에서 화면이 깨진다.
- 커밋 전에 빌드나 테스트를 돌리지 않고 완료했다고 말한다.

사람이 직접 코딩할 때도 이런 실수는 생긴다.

그런데 AI는 속도가 빠르기 때문에, 실수를 발견했을 때 이미 변경 범위가 너무 커져 있을 수 있다.

그래서 바이브코딩에는 “작업을 맡기는 기술”보다 “작업을 제한하는 구조”가 더 중요하다고 생각했다.

## 기획이 먼저다

수업에서 가장 인상적이었던 부분은 “기획이 가장 중요하다”는 말이었다.

AI에게 바로 “이 기능 만들어줘”라고 말하기보다, 먼저 계획을 여러 번 다듬는다.

흐름은 대략 이렇다.

1. `/clear`로 컨텍스트를 비운다.
2. plan mode로 들어간다.
3. 만들고 싶은 아이디어와 사용하고 싶은 기술을 넣는다.
4. 1차 플랜을 받는다.
5. 더 상세하게 요구한다.
6. 다시 플랜을 받는다.
7. 아키텍처와 기술적인 질문을 한다.
8. 괜찮다고 생각될 때까지 여러 번 반복한다.

수업에서는 보통 5번 이상 플랜을 반복한다고 했다.

처음에는 “그렇게까지 해야 하나?”라는 생각이 들었다.

그런데 생각해보면 AI가 코드를 잘못 짜는 이유 중 하나는 처음 계획이 흐릿하기 때문이다.

사람도 요구사항이 흐릿하면 이상한 방향으로 구현한다. AI도 마찬가지다.

그래서 기획 단계에서는 코드를 빨리 받는 것보다, 만들지 않을 것과 반드시 지켜야 할 기준을 정하는 것이 더 중요하다.

## 문서 3종으로 기준 만들기

하네스 구조에서는 구현 전에 `docs/` 아래 문서를 먼저 만든다.

기본 문서는 3개다.

```text
docs/PRD.md
docs/ARCHITECTURE.md
docs/ADR.md
```

프론트엔드가 있다면 여기에 하나를 더 추가한다.

```text
docs/UI_GUIDE.md
```

각 문서의 역할은 다르다.

`PRD.md`는 무엇을 만들지 정하는 문서다.

여기에는 프로젝트명, 목표, 사용자, 핵심 기능, MVP 범위, 제외할 상황이 들어간다.

`ARCHITECTURE.md`는 어떻게 만들지 정하는 문서다.

기술 스택, 디렉토리 구조, 데이터 흐름, API 구조, 테스트 전략 같은 내용을 정리한다.

`ADR.md`는 왜 그렇게 만들기로 했는지 남기는 문서다.

기술 선택은 항상 트레이드오프가 있다. 그래서 “무엇을 선택했는가”뿐 아니라 “왜 선택했는가”, “무엇을 포기했는가”까지 남긴다.

프론트엔드가 있다면 `UI_GUIDE.md`도 중요하다.

AI가 UI를 만들 때 흔히 보라색 그라데이션, 둥근 카드, 빛나는 효과 같은 전형적인 패턴으로 가는 경우가 있다. 이런 것을 막으려면 디자인 원칙과 금지할 UI 패턴을 미리 적어두는 것이 좋다.

## AGENTS.md는 프로젝트의 작업 규칙이다

Claude에서는 `CLAUDE.md`를 사용하지만, Codex에서는 `AGENTS.md`를 기준으로 작업 규칙을 정리한다.

`AGENTS.md`에는 이런 내용이 들어가면 좋다.

- 프로젝트 목적
- 중요한 폴더 구조
- 사용 기술
- 아키텍처 규칙
- TDD 원칙
- lint, build, test 명령어
- 커밋 규칙
- 금지 사항
- 변경 전 반드시 읽어야 할 문서

예를 들어 새 기능을 만들 때는 테스트를 먼저 작성하라고 적을 수 있다.

하지만 여기서 중요한 점이 있다.

`AGENTS.md`나 markdown 파일에 적힌 규칙은 “명령”이지 “강제”는 아니다.

AI가 대부분 지켜주기는 하지만, 상황에 따라 빼먹을 수도 있다.

그래서 정말 강제해야 하는 것은 hook으로 옮겨야 한다.

## hook으로 강제하기

수업 내용을 Codex 기준으로 정리하면서 가장 중요하다고 느낀 부분이 hook이다.

hook은 특정 시점에 스크립트를 실행해서 작업 흐름을 강제하는 장치다.

예를 들어 TDD를 지키고 싶다면 `tdd-guard.sh` 같은 hook을 둘 수 있다.

```text
.codex/hooks/tdd-guard.sh
```

이 hook의 역할은 구현 전에 테스트가 있는지, 또는 테스트 계획이 있는지 확인하는 것이다.

또 커밋 전에 검증을 강제하려면 이런 hook을 둘 수 있다.

```text
.codex/hooks/pre-commit-verify.sh
```

이 hook에서는 프로젝트에 맞게 아래 커맨드를 실행한다.

```bash
npm run lint
npm run build
npm test
```

Spring/Gradle 프로젝트라면 이렇게 바뀔 수 있다.

```bash
./gradlew test
./gradlew build
```

내가 이해한 hook의 이유는 단순하다.

반드시 지켜야 하는 것은 문서에만 적어두면 부족하다.

실제로 실행되는 스크립트로 막아야 한다.

## 작업은 phase와 step으로 나눈다

하네스의 핵심은 작업을 아주 작게 나누는 것이다.

예를 들어 “게시판 만들기”를 한 번에 맡기지 않는다.

대신 이런 식으로 나눈다.

```text
phases/
├── index.json
└── 0-mvp/
    ├── index.json
    ├── step0.md
    ├── step1.md
    └── step2.md
```

`phases/index.json`은 전체 phase 상태를 관리한다.

`phases/0-mvp/index.json`은 해당 phase 안의 step 상태를 관리한다.

각 step 파일은 실제 작업 지시서다.

중요한 점은 step 파일이 독립적이어야 한다는 것이다.

“아까 말한 것처럼” 같은 표현을 쓰면 안 된다.

새로운 Codex 세션이 step 파일만 읽어도 무엇을 해야 하는지 알 수 있어야 한다.

그래서 step 파일에는 읽어야 할 문서, 수정할 파일, 함수나 클래스 시그니처, Acceptance Criteria, 금지사항을 모두 적는다.

## Acceptance Criteria는 실행 가능한 커맨드여야 한다

좋은 step 파일에는 추상적인 완료 조건보다 실행 가능한 검증 커맨드가 들어가야 한다.

예를 들어 이렇게 적는 것이 좋다.

```bash
npm run build
npm test
```

또는 프로젝트에 따라 이렇게 쓸 수 있다.

```bash
./gradlew test
./gradlew build
```

“정상 동작해야 한다”는 말은 너무 추상적이다.

AI가 완료했다고 말해도 실제로 검증했는지 알기 어렵다.

반대로 실행 커맨드가 있으면 성공과 실패가 분명해진다.

## 화면은 브라우저로 직접 확인해야 한다

프론트엔드 작업에서는 lint, build, test만으로 부족할 때가 많다.

빌드는 성공했는데 실제 화면은 깨질 수 있다.

버튼은 보이는데 클릭이 안 될 수도 있다.

데스크톱에서는 괜찮은데 모바일에서는 텍스트가 겹칠 수도 있다.

그래서 UI가 있는 step에는 브라우저 테스트를 포함해야 한다.

예를 들어 Codex에게 이렇게 시킬 수 있다.

```text
브라우저로 http://localhost:3000에 접속해서 핵심 사용자 흐름을 테스트해줘.

확인할 것:
- 첫 화면이 정상 렌더링되는가
- 주요 버튼이 클릭되는가
- 폼 입력과 제출이 동작하는가
- 페이지 이동 또는 라우팅이 정상인가
- 모바일 viewport에서도 레이아웃이 깨지지 않는가
- 콘솔 에러가 없는가
- 네트워크 요청 실패가 없는가
```

내가 생각하는 최소 브라우저 QA 체크리스트는 이렇다.

- 화면이 빈 페이지로 뜨지 않는가?
- 콘솔 에러가 없는가?
- 주요 버튼이 실제로 동작하는가?
- 폼 입력과 제출이 되는가?
- API 요청 실패가 없는가?
- 모바일에서 텍스트와 버튼이 겹치지 않는가?
- 핵심 사용자 흐름을 끝까지 수행할 수 있는가?

브라우저 테스트는 자동 테스트를 대체하는 것이 아니다.

자동 테스트를 통과한 뒤, 실제 사용자의 관점에서 한 번 더 확인하는 단계다.

## execute.py는 하네스를 실행하는 장치다

수업 레포에는 `scripts/execute.py`가 있었다.

역할은 phase 안의 step을 순서대로 실행하는 것이다.

대략 이런 일을 한다.

- `feat-{task-name}` 브랜치를 만든다.
- `CLAUDE.md`와 `docs/*.md`를 step 프롬프트에 넣는다.
- 이전 step의 summary를 다음 step에 전달한다.
- 실패하면 최대 3회 재시도한다.
- step 상태를 `completed`, `error`, `blocked`로 업데이트한다.
- 코드 변경 커밋과 메타데이터 커밋을 분리한다.
- 필요하면 push까지 한다.

Codex로 옮길 때는 여기서 몇 가지를 바꿔야 한다.

`CLAUDE.md`는 `AGENTS.md`로 바꿔야 한다.

그리고 내부에서 Claude CLI를 호출하는 부분은 Codex CLI 실행 방식에 맞게 바꿔야 한다.

즉, 수업 레포는 그대로 복사하는 것이 아니라 Codex용으로 포팅해야 한다.

## review도 구조화해야 한다

구현이 끝난 뒤에도 그냥 “리뷰해줘”라고 하면 결과가 들쭉날쭉할 수 있다.

그래서 review skill이나 `AGENTS.md`에 체크리스트를 넣어두는 것이 좋다.

체크리스트는 이런 식으로 만들 수 있다.

- `ARCHITECTURE.md`의 디렉토리 구조를 지켰는가?
- `ADR.md`에 적힌 기술 결정을 벗어나지 않았는가?
- 새 기능에 대한 테스트가 있는가?
- `AGENTS.md`의 CRITICAL 규칙을 위반하지 않았는가?
- 빌드가 통과하는가?
- 브라우저 테스트를 했는가?
- 콘솔 에러나 네트워크 실패가 없는가?

이렇게 리뷰 기준을 고정해두면 AI가 단순히 코드 스타일만 보는 것이 아니라, 프로젝트의 기준에 맞게 변경 사항을 확인하게 된다.

## 내가 헷갈렸던 부분

처음에는 바이브코딩을 “AI에게 많이 맡길수록 좋은 방식”이라고 생각했다.

하지만 수업을 듣고 나서는 반대로 이해하게 됐다.

좋은 바이브코딩은 많이 맡기는 것이 아니라 잘게 맡기는 것이다.

AI가 한 번에 수정할 수 있는 범위를 줄이고, 각 단계마다 검증 기준을 넣고, 실패하면 멈추게 해야 한다.

또 하나 헷갈렸던 부분은 markdown 명령과 hook의 차이다.

처음에는 `AGENTS.md`에 “TDD를 지켜라”라고 적으면 충분하다고 생각했다.

하지만 markdown은 어디까지나 지시다.

진짜 강제하려면 hook이 필요하다.

## 실제로 사용하면 좋은 상황

이 하네스 방식은 특히 이런 상황에서 좋아 보인다.

- 새 프로젝트를 AI와 함께 빠르게 만들 때
- MVP를 만들지만 품질 기준은 놓치고 싶지 않을 때
- 여러 step으로 나눠서 안정적으로 구현하고 싶을 때
- 프론트엔드 화면 검증까지 포함하고 싶을 때
- 나중에 왜 이런 기술 결정을 했는지 남겨야 할 때
- AI가 마음대로 범위를 키우는 것을 막고 싶을 때

반대로 아주 작은 수정에는 너무 무거울 수 있다.

오타 수정이나 간단한 문구 변경까지 phase를 만들 필요는 없다.

하지만 기능 단위 작업, 아키텍처 변경, 인증, 결제, 데이터 모델링처럼 위험도가 있는 작업에는 하네스 방식이 훨씬 안전해 보인다.

## 정리

이번 수업을 정리하면서 바이브코딩에 대한 생각이 바뀌었다.

바이브코딩은 그냥 분위기와 감으로 코드를 만드는 방식이 아니다.

오히려 좋은 바이브코딩을 하려면 기획, 문서, step 분리, TDD, hook, 브라우저 테스트 같은 장치가 필요하다.

AI는 빠르다.

그래서 더더욱 작업 범위를 작게 나누고, 실패했을 때 멈추게 하고, 검증 커맨드를 강제해야 한다.

결국 중요한 것은 AI에게 코드를 쓰게 만드는 것이 아니라, AI가 좋은 개발 흐름 안에서 코드를 쓰게 만드는 것이다.
