# [FaithLog 개발기 1] 교회 공동체 운영을 앱으로 관리해보려는 이유

## 들어가며

이번에 개인 프로젝트로 **FaithLog**라는 서비스를 기획하게 되었다.

FaithLog는 교회 공동체 안에서 반복적으로 발생하는 경건생활 체크, 예배 참여 투표, 커피 주문, 벌금 납부 상태를 관리하는 모바일 앱이다.

처음부터 거창한 서비스를 만들려고 한 것은 아니었다.
실제 공동체 안에서 매주 반복되는 운영 업무를 보면서, 이 과정을 조금 더 편하게 만들 수 있지 않을까 하는 생각에서 시작했다.

현재는 많은 내용이 카카오톡, 수기 기록, 사람 간 확인으로 이루어진다.
물론 규모가 작을 때는 이 방식도 충분히 가능하다.
하지만 인원이 늘어나고 매주 반복되는 항목이 많아지면 관리자의 부담이 커지고, 누락이나 착오도 생길 수 있다.

그래서 이번 프로젝트에서는 실제 운영 상황을 기준으로 문제를 정의하고, MVP 범위를 좁혀서 하나씩 구현해보려고 한다.

---

## 현재 운영 방식에서 느낀 문제

공동체 안에서는 매주 반복되는 일이 많다.

예를 들면 다음과 같다.

* 경건생활 체크
* 수요예배 참여 여부 확인
* 토요 목자모임 참여 여부 확인
* 커피 주문
* 벌금 또는 커피값 납부 확인
* 미체크자, 미응답자, 미납자 알림

이런 일들은 대부분 카카오톡 메시지, 투표, 수기 기록, 담당자의 확인으로 이루어진다.

처음에는 단순해 보이지만, 실제로 운영해보면 몇 가지 불편함이 생긴다.

첫 번째는 **정보가 흩어진다**는 점이다.

경건생활 체크는 따로 확인하고, 예배 참석 여부는 카카오톡 투표로 확인하고, 커피 주문은 또 다른 메시지로 확인하는 식이다.
그러다 보니 특정 사람이 체크를 했는지, 투표를 했는지, 돈을 냈는지 확인하려면 여러 내용을 다시 찾아봐야 한다.

두 번째는 **담당자의 반복 업무가 많다**는 점이다.

매주 누가 입력하지 않았는지 확인하고, 누가 투표하지 않았는지 확인하고, 누가 납부하지 않았는지 다시 알려주는 일이 반복된다.

세 번째는 **사용자 입장에서도 확인이 어렵다**는 점이다.

내가 이번 주에 경건생활을 몇 번 했는지, 벌금이 얼마인지, 커피값을 냈는지 한눈에 확인하기 어렵다.
결국 담당자에게 다시 물어보거나, 이전 카톡 내용을 찾아봐야 한다.

이런 문제를 보면서 “이 정도는 앱으로 관리할 수 있지 않을까?”라는 생각이 들었다.

## 만들고 싶은 서비스

FaithLog의 목표는 공동체 운영을 완전히 자동화하는 것이 아니다.

오히려 목표는 단순하다.

> 반복되는 확인 업무를 줄이고, 구성원이 자신의 상태를 쉽게 확인할 수 있게 만드는 것

처음부터 모든 기능을 넣기보다는, 실제로 필요한 핵심 기능만 먼저 구현하려고 한다.

초기 MVP에서는 다음 기능을 중심으로 개발한다.

* 회원가입 / 로그인
* 그룹 가입
* 경건생활 하루별 체크
* 주간 경건생활 제출
* 벌금 자동 계산
* 수요예배 참여 투표
* 토요 목자모임 참여 투표
* 커피 주문 투표
* 투표 현황 조회
* 벌금/커피값 청구 항목 생성
* 사용자의 납부 완료 처리
* FCM 푸시 알림

## 왜 점심 기능은 제외했나

처음에는 점심 투표와 점심 정산도 넣으려고 했다.

예를 들어 토요일에 김치찜, 지코바, 치킨 같은 음식을 함께 주문하고, 사람별로 금액을 나누는 기능을 생각했다.

하지만 설계를 하다 보니 점심은 생각보다 변수가 많았다.

* 총 주문 금액이 매번 다르다.
* 먹는 사람이 중간에 바뀔 수 있다.
* 사람마다 먹는 양이 다를 수 있다.
* 동일 분배가 아닐 수도 있다.
* 담당자가 직접 금액을 조정해야 할 수 있다.
* 주문 후 취소나 추가 주문이 생길 수 있다.

이 기능까지 MVP에 넣으면 프로젝트의 핵심이 흐려질 것 같았다.

그래서 이번 MVP에서는 점심 기능을 제외하고, **경건생활, 투표, 커피 주문, 납부 관리**에 집중하기로 했다.

점심 기능은 나중에 서비스가 안정화된 뒤 확장 기능으로 추가하는 것이 더 현실적이라고 판단했다.

## 핵심 기능 1: 경건생활 체크

경건생활 체크는 이 서비스의 핵심 기능 중 하나다.

처음에는 사용자가 한 주 동안 큐티를 몇 번 했는지, 기도를 몇 번 했는지, 말씀을 몇 번 읽었는지를 숫자로 입력하는 방식도 생각했다.

하지만 사용자 입장에서는 숫자만 입력하는 것보다, 하루씩 체크하는 방식이 더 직관적이라고 생각했다.

그래서 경건생활은 다음과 같은 방식으로 설계했다.

* 사용자는 캘린더에서 날짜를 선택한다.
* 날짜별로 큐티, 기도, 말씀읽기를 체크한다.
* 주간 제출 시 하루별 체크 기록을 합산한다.
* 합산 결과를 기준으로 벌금을 계산한다.

예를 들면 이런 흐름이다.

```text
월요일: 큐티 O, 기도 O, 말씀 X
화요일: 큐티 O, 기도 X, 말씀 O
수요일: 큐티 X, 기도 O, 말씀 O
...
```

이렇게 하면 사용자는 자신이 어느 날 무엇을 했는지 확인할 수 있고, 프론트에서는 캘린더 형식으로 자연스럽게 보여줄 수 있다.

---

## 핵심 기능 2: 벌금 계산

벌금은 경건생활 체크를 기준으로 계산한다.

현재 생각한 규칙은 다음과 같다.

| 항목         |    기준 |                  벌금 |
| ---------- | ----: | ------------------: |
| 큐티         |  주 5회 |         부족 1회당 500원 |
| 기도         |  주 5회 |         부족 1회당 500원 |
| 말씀읽기       |  주 5회 |         부족 1회당 300원 |
| 토요 목자모임 지각 | 지각 시간 | 기본 1,000원 + 분당 100원 |

중요한 점은 벌금을 항목별로 따로 청구하지 않는다는 것이다.

예를 들어 큐티 2회 부족, 토요 목자모임 5분 지각이 있다면 청구 항목을 이렇게 여러 개 만들지 않는다.

```text
큐티 2회 부족 1,000원
토요 목자모임 지각 1,500원
```

대신 한 주의 벌금을 합산해서 하나의 청구 항목으로 만든다.

```text
2026년 6월 2주차 벌금 2,500원
```

이렇게 하면 사용자는 벌금을 한 번에 확인하고 납부할 수 있다.

세부 사유는 청구 항목의 reason 값에 요약해서 남긴다.

```text
큐티 2회 부족, 지각 5분
```

## 핵심 기능 3: 수요예배/토요 목자모임 투표

공동체 안에서는 매주 반복되는 투표가 있다.

대표적으로 수요예배 참여 여부와 토요 목자모임 참여 여부가 있다.

이런 투표는 매주 같은 형식으로 반복되기 때문에, 반복 투표 템플릿을 두고 자동으로 생성할 수 있도록 설계했다.

예를 들어 수요예배 투표는 다음과 같은 선택지를 가질 수 있다.

```text
참여
미참여
미정
```

토요 목자모임 투표는 다음과 같이 구성할 수 있다.

```text
정시 참석
지각 예정
미정
```

투표 현황도 확인할 수 있어야 한다.

관리자는 다음 정보를 볼 수 있다.

* 선택지별 응답 수
* 사용자별 응답 내역
* 미응답자 목록
* 전체 응답률

이 기능은 나중에 FCM 알림과 연결된다.

예를 들어 투표 마감 전까지 응답하지 않은 사람에게만 알림을 보낼 수 있다.

## 핵심 기능 4: 커피 주문

MVP에서는 점심 기능은 제외하지만, 커피 주문은 포함하기로 했다.

커피는 점심보다 구조가 단순하다.

각 메뉴마다 가격이 정해져 있고, 사용자가 하나를 선택하면 해당 가격으로 청구 항목을 만들 수 있기 때문이다.

예를 들어 사용자가 커피 투표에서 아이스 아메리카노를 선택하면 다음과 같은 청구 항목이 생성된다.

```text
COFFEE / 아이스 아메리카노 / 1,800원 / 미납
```

사용자가 “안 마셔요”를 선택하면 가격이 0원이므로 청구 항목을 생성하지 않는다.

커피 메뉴는 DB enum으로 관리하지 않고, Spring enum으로 관리할 예정이다.

DB에는 다음 정보만 저장한다.

* compose_menu_code
* content
* price_amount

여기서 중요한 점은 실제 청구 금액은 Spring enum이 아니라 `poll_options.price_amount`를 기준으로 한다는 것이다.

Spring enum의 가격은 기본값일 뿐이고, 투표 생성 시점에 가격을 복사해서 저장한다.
그래야 나중에 메뉴 가격이 바뀌어도 과거 주문 금액이 변하지 않는다.

## 핵심 기능 5: 납부 완료 처리

이 서비스에서는 별도의 결제 API를 붙이지 않는다.

또한 입금 인증 사진도 받지 않는다.

공동체 내부에서 사용하는 서비스이고, 기본적으로 양심적으로 사용하는 것을 전제로 한다.

그래서 사용자는 실제 계좌이체를 한 뒤 앱에서 “납부했어요”를 누른다.

그러면 해당 청구 항목은 바로 납부 완료 상태가 된다.

```text
UNPAID → PAID
```

관리자 승인 과정은 MVP에서 제외한다.

물론 더 엄격한 관리가 필요해지면 나중에 승인 상태를 추가할 수 있지만, 초기에는 사용성을 우선으로 두기로 했다.

## 핵심 기능 6: 알림

배치는 정산 생성이 아니라 알림 발송 용도로만 사용한다.

예를 들어 다음과 같은 알림을 보낼 수 있다.

* 일요일 경건생활 체크 알림
* 월요일 미체크자 알림
* 수요예배 투표 시작 알림
* 수요예배 투표 미응답자 알림
* 토요 목자모임 투표 시작 알림
* 토요 목자모임 투표 미응답자 알림
* 커피 투표 시작 알림
* 커피 투표 미응답자 알림
* 미납 항목 알림

알림은 FCM을 사용할 예정이다.

사용자의 FCM 토큰은 별도 테이블에서 관리하고, 한 사용자가 여러 기기를 사용할 수 있도록 설계할 예정이다.

## 이번 프로젝트에서 얻고 싶은 것

이번 프로젝트를 통해 단순히 CRUD만 구현하는 것이 아니라, 실제 운영 상황을 고려한 백엔드 설계를 해보고 싶다.

특히 다음 부분을 중점적으로 경험하려고 한다.

* 권한과 담당자 분리
* 반복 투표 설계
* 미응답자 조회
* 하루별 체크와 주간 집계 구조
* 벌금 계산 로직
* 청구 항목 생성과 납부 상태 관리
* FCM 알림 발송
* 실제 운영을 고려한 MVP 범위 조절

처음에는 넣고 싶은 기능이 많았다.

하지만 설계를 하다 보니 모든 기능을 처음부터 넣는 것보다, 핵심 기능을 명확히 정하고 완성하는 것이 더 중요하다는 생각이 들었다.

그래서 이번 MVP에서는 점심 기능을 과감히 제외하고, 경건생활, 투표, 커피 주문, 납부 관리에 집중하려고 한다.
