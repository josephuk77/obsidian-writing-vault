
# FaithLog 개발기 API 설계: 운영 흐름을 API 구조로 옮기기

## 들어가며

이전 글에서는 FaithLog의 ERD를 설계하면서 실제 운영 흐름을 데이터 구조로 옮기는 과정을 정리했다.

이번 글에서는 그 ERD를 바탕으로 API를 어떻게 설계했는지 정리해보려고 한다.

처음에는 API 설계를 단순하게 생각했다.

사용자 API, 캠퍼스 API, 경건생활 API, 투표 API, 벌금 API 정도로 나누면 될 것 같았다.

하지만 ERD를 설계하면서 실제 운영 흐름을 더 자세히 생각해보니 API도 단순 CRUD로만 만들기에는 부족한 부분이 많았다.

특히 다음 부분을 많이 고민했다.

- 서비스 전체 권한과 캠퍼스 내부 역할을 API에서 어떻게 구분할 것인가
    
- 사용자가 여러 캠퍼스에 속할 수 있는 구조를 어떻게 표현할 것인가
    
- 경건생활을 매일 체크하는 경우와 일요일에 한 번에 제출하는 경우를 모두 어떻게 지원할 것인가
    
- 벌금과 커피비를 같은 납부 흐름으로 처리할 수 있을 것인가
    
- 투표 결과뿐 아니라 미참여자 조회와 알림까지 고려할 수 있는 구조인가
    
- 실제 운영할 서비스인데 API 명세를 어느 수준까지 공개해도 되는가
    

이번 글에서는 전체 API 명세를 모두 공개하기보다, 실제 운영에 문제가 될 수 있는 세부 정보는 제외하고 API를 설계하면서 고민한 흐름을 중심으로 정리하려고 한다.

---

## 공개하지 않기로 한 내용

이번 프로젝트는 단순 포트폴리오용이 아니라 실제 운영을 목표로 하고 있다.

그래서 API 설계 글을 작성할 때도 전체 명세를 그대로 공개하지 않기로 했다.

API 명세 자체가 곧바로 보안 취약점이 되는 것은 아니지만, 운영 서비스에서는 너무 자세한 API 구조가 외부에 공개될 필요는 없다고 판단했다.

블로그에서 제외한 내용은 다음과 같다.

- 전체 API 목록
    
- 실제 운영 도메인
    
- 실제 초대코드 형식
    
- 실제 계좌 정보
    
- 실제 사용자 이름과 이메일
    
- 관리자 권한 변경 API의 상세 경로
    
- 납부 상태 변경 API의 상세 조건
    
- 토큰 만료 시간과 내부 검증 로직
    
- FCM 발송 대상 조회 로직 전체
    
- 내부 스케줄러 API
    
- 보안과 관련된 세부 예외 처리
    

대신 이번 글에서는 “어떤 문제를 해결하기 위해 어떤 API 구조를 선택했는가”에 집중하려고 한다.

## API 설계 기준

FaithLog의 API는 캠퍼스를 중심으로 설계했다.

FaithLog에서 관리되는 대부분의 데이터는 특정 캠퍼스에 속한다.

경건생활 기록도 캠퍼스 기준이고, 벌금 규칙도 캠퍼스 기준이다.

투표, 커피 담당, 납부 계좌, 알림도 모두 캠퍼스 단위로 관리된다.

그래서 API URL도 캠퍼스를 기준으로 잡았다.

```http
/campuses/{campusId}/...
```

사용자 본인의 데이터를 조회하거나 수정하는 경우에는 `me`를 사용했다.

```http
/campuses/{campusId}/devotions/me/...
/campuses/{campusId}/charges/me/...
```

이렇게 설계하면 프론트에서는 현재 선택한 캠퍼스를 기준으로 API를 호출할 수 있다.

백엔드에서도 요청을 처리하기 전에 사용자가 해당 캠퍼스에 속한 사람인지 먼저 검증할 수 있다.

즉, API 경로 자체가 권한 검증의 기준을 자연스럽게 드러내도록 설계했다.

## 인증 API 설계

인증은 Access Token과 Refresh Token을 사용하는 방식으로 설계했다.

처음에는 로그인 응답 body에 토큰을 담아 내려주는 방식을 생각했다.

하지만 실제 운영을 고려하면 프론트엔드 JavaScript에서 토큰에 직접 접근하는 상황을 줄이는 것이 더 낫다고 판단했다.

그래서 Access Token과 Refresh Token은 HttpOnly Cookie로 내려주는 방향으로 설계했다.

인증 API의 큰 흐름은 다음과 같다.

|API|역할|
|---|---|
|회원가입|사용자 계정 생성|
|로그인|Access Token, Refresh Token 발급|
|토큰 재발급|Refresh Token 기반 Access Token 재발급|
|로그아웃|Refresh Token 폐기 및 Cookie 삭제|
|내 정보 조회|로그인 사용자 정보와 소속 캠퍼스 목록 조회|

Refresh Token은 일반 API 요청에 매번 포함될 필요가 없다.

그래서 Refresh Token Cookie는 토큰 재발급 요청에만 전송되도록 범위를 제한하는 방향으로 설계했다.

또한 서버에는 Refresh Token 원본을 저장하지 않고 해시값을 저장하도록 했다.

운영 서비스이기 때문에 구체적인 만료 시간이나 내부 검증 방식은 공개하지 않는다.

## 사용자와 캠퍼스 역할 구조

처음에는 캠퍼스 멤버의 역할만으로 모든 권한을 처리하려고 했다.

예를 들어 캠퍼스 멤버 역할을 `OWNER`, `ADMIN`, `MEMBER` 정도로 두고 권한을 구분하는 방식이다.

하지만 설계하다 보니 서비스 전체 권한과 캠퍼스 내부 역할이 섞이는 문제가 있었다.

서비스 전체에서 캠퍼스를 생성할 수 있는 사람과, 특정 캠퍼스 안에서 운영을 담당하는 사람은 역할의 범위가 다르다.

그래서 권한을 두 가지로 나누었다.

### 서비스 전체 권한

|권한|설명|
|---|---|
|USER|일반 사용자|
|MANAGER|캠퍼스 생성 가능|
|ADMIN|전체 관리자|

서비스 전체 권한은 `users` 기준으로 관리한다.

예를 들어 캠퍼스를 생성할 수 있는지, 사용자 권한을 변경할 수 있는지는 서비스 전체 권한으로 판단한다.

### 캠퍼스 내부 역할

|역할|설명|
|---|---|
|MINISTER|전도사님|
|ELDER|엘더|
|CAMPUS_LEADER|캠장|
|MEMBER|일반 캠퍼스원|

캠퍼스 내부 역할은 `campus_members` 기준으로 관리한다.

한 사용자가 여러 캠퍼스에 속할 수 있기 때문이다.

예를 들어 어떤 사용자는 A캠퍼스에서는 캠장이고, B캠퍼스에서는 일반 멤버일 수 있다.

그래서 사용자 자체의 권한과 캠퍼스 안에서의 역할을 분리하는 것이 더 자연스럽다고 판단했다.

## 캠퍼스 API 설계

캠퍼스는 관리자가 직접 생성할 수 있도록 설계했다.

처음에는 초대코드를 통해 특정 권한을 가진 사용자가 자동으로 캠퍼스를 만들 수 있게 할까 고민했다.

하지만 실제 운영에서는 아무나 캠퍼스를 생성할 수 있으면 관리가 어려워질 수 있다.

그래서 캠퍼스 생성은 서비스 전체 권한을 가진 사용자만 가능하게 했다.

|기능|설명|
|---|---|
|캠퍼스 생성|관리자 또는 매니저가 캠퍼스를 생성|
|캠퍼스 목록 조회|캠퍼스 이름, 지역 기준으로 조회|
|내 캠퍼스 목록 조회|내가 속한 캠퍼스 목록 조회|
|캠퍼스 상세 조회|캠퍼스 기본 정보 조회|
|캠퍼스 수정|권한 있는 사용자만 수정|
|초대코드 재발급|캠퍼스 운영자가 초대코드 재발급|

캠퍼스에는 `name`과 `region`을 분리해서 두었다.

예를 들어 `region`은 분당, 수지 같은 지역 값이고, `name`은 1캠, 2캠 같은 캠퍼스 이름이다.

처음에는 분당과 수지를 캠퍼스 이름으로 직접 사용할까 생각했지만, 실제 운영에서는 같은 지역 안에서도 여러 캠퍼스가 있을 수 있다.

그래서 지역과 캠퍼스 이름을 분리하는 것이 더 확장성 있다고 판단했다.

## 캠퍼스 가입 API 설계

캠퍼스 가입 방식도 고민이 있었다.

처음에는 사용자가 가입 요청을 보내고, 관리자가 승인하는 구조를 생각했다.

하지만 MVP 단계에서는 승인 과정이 오히려 운영을 복잡하게 만들 수 있다고 판단했다.

그래서 초대코드 기반 가입 방식을 선택했다.

흐름은 다음과 같다.

- 사용자는 누구나 회원가입할 수 있다.
    
- 회원가입한 사용자는 기본적으로 일반 사용자 권한을 가진다.
    
- 캠퍼스에 들어가려면 초대코드를 입력한다.
    
- 초대코드가 유효하면 해당 캠퍼스에 바로 가입된다.
    
- 기본 캠퍼스 역할은 일반 멤버로 등록된다.
    

이 방식은 가입 승인 방식보다 단순하고, 초대코드를 알고 있는 사람만 캠퍼스에 들어올 수 있다는 점에서 초기 운영에 적합하다고 판단했다.

## 캠퍼스 멤버 API 설계

캠퍼스 멤버 API는 단순히 사용자 목록을 조회하는 기능이 아니다.

FaithLog에서는 같은 사용자가 여러 캠퍼스에 속할 수 있고, 캠퍼스마다 다른 역할을 가질 수 있다.

그래서 회원 조회도 `users`만 보는 것이 아니라 `campus_members`와 `campuses`를 함께 조회하는 방식으로 설계했다.

관리자가 멤버를 조회할 때는 다음 기준이 필요하다.

|조회 기준|설명|
|---|---|
|캠퍼스 이름|1캠, 2캠 등|
|지역|분당, 수지 등|
|캠퍼스 역할|전도사님, 엘더, 캠장, 일반 멤버|
|상태|활성, 비활성|
|검색어|이름 또는 이메일|

캠퍼스 내부 역할 변경은 서비스 전체 관리자뿐 아니라 해당 캠퍼스의 운영 역할을 가진 사용자도 할 수 있도록 설계했다.

예를 들어 전도사님, 엘더, 캠장은 해당 캠퍼스 안에서 구성원의 역할을 변경하거나 커피 담당자를 지정할 수 있다.

반면 서비스 전체 권한 변경은 전체 관리자만 가능하게 분리했다.

이렇게 분리하면 전역 관리자 권한과 캠퍼스 운영 권한이 섞이지 않는다.

## 커피 담당 API 설계

커피 담당은 처음에 역할로 넣을까 고민했다.

예를 들어 캠퍼스 역할에 `COFFEE`를 추가하는 방식이다.

하지만 이렇게 하면 문제가 생긴다.

어떤 사용자는 일반 멤버이면서 커피 담당일 수 있다.

또 어떤 사용자는 캠장이면서 커피 담당일 수도 있다.

즉, 캠퍼스 역할은 권한이나 직분에 가깝고, 커피 담당은 운영 담당에 가깝다.

그래서 커피 담당은 캠퍼스 역할과 분리해서 별도의 담당자 지정 구조로 설계했다.

|기능|설명|
|---|---|
|커피 담당 조회|현재 활성화된 커피 담당자 조회|
|커피 담당 등록|권한 있는 사용자가 커피 담당자 지정|
|커피 담당 해제|더 이상 담당자가 아닌 경우 해제|

처음에는 커피 담당에 시작일과 종료일을 둘까 고민했다.

하지만 실제 운영에서는 담당 기간이 항상 규칙적으로 정해지는 것은 아니다.

그래서 기간을 기준으로 자동 관리하기보다, 관리자가 현재 담당자를 직접 지정하고 해제하는 방식이 더 현실적이라고 판단했다.

## 경건생활 API 설계

경건생활 체크는 API 설계에서 가장 많이 고민한 부분이다.

처음에는 한 주 단위로 큐티 몇 회, 기도 몇 회, 말씀 몇 회만 저장하면 된다고 생각했다.

이 방식은 벌금 계산에는 편하다.

하지만 사용자 입장에서는 내가 어느 날 체크했는지 확인하기 어렵다.

또 프론트에서 캘린더 형식으로 보여주려면 하루별 기록이 필요하다.

그래서 경건생활 API는 두 가지 흐름을 모두 지원하도록 설계했다.

|API 흐름|설명|
|---|---|
|하루 단위 체크|사용자가 매일 경건생활을 체크|
|주간 일괄 제출|사용자가 한 주치를 한 번에 제출|

---

## 하루 단위 체크

하루 단위 체크는 사용자가 매일 경건생활을 기록할 때 사용한다.

대표적인 API 형태는 다음과 같다.

```http
PUT /campuses/{campusId}/devotions/me/days/{recordDate}
```

요청 값은 날짜별 체크 여부를 담는다.

```json
{
  "quietTimeChecked": true,
  "prayerChecked": true,
  "bibleReadingChecked": false
}
```

이 API는 해당 날짜의 체크 정보를 저장한다.

만약 해당 날짜가 속한 주차의 주간 기록이 없다면 서버에서 주간 기록을 먼저 생성한다.

다만 하루 단위 체크에서는 최종 제출 처리를 하지 않는다.

월요일에 체크했다고 해서 한 주의 경건생활 제출이 완료되었다고 볼 수는 없기 때문이다.

그래서 하루 단위 체크 API는 다음 역할만 담당한다.

- 날짜 기준으로 주차 계산
    
- 주간 기록이 없으면 생성
    
- 해당 날짜의 체크 기록 생성 또는 수정
    
- 주간 카운트 재계산
    
- 최종 제출 시간은 변경하지 않음
    
- 벌금 항목은 생성하지 않음
    

---

## 주간 일괄 제출

주간 일괄 제출은 사용자가 일요일에 한 주의 경건생활을 한 번에 제출하는 경우를 고려한 API다.

대표적인 API 형태는 다음과 같다.

```http
PUT /campuses/{campusId}/devotions/me/weeks/{weekStartDate}
```

요청 값은 한 주의 날짜별 체크 목록과 토요 지각 시간을 포함한다.

```json
{
  "dailyChecks": [
    {
      "recordDate": "2026-06-08",
      "quietTimeChecked": true,
      "prayerChecked": true,
      "bibleReadingChecked": false
    },
    {
      "recordDate": "2026-06-09",
      "quietTimeChecked": true,
      "prayerChecked": false,
      "bibleReadingChecked": true
    }
  ],
  "saturdayLateMinutes": 5,
  "submit": true
}
```

이 API의 핵심은 한 번에 제출하더라도 날짜별 기록이 생성된다는 점이다.

요청에 일부 날짜만 포함되어 있어도 서버에서는 해당 주차의 날짜별 row를 모두 생성한다.

요청에 없는 날짜는 체크하지 않은 값으로 처리한다.

이렇게 설계한 이유는 다음과 같다.

- 사용자가 매일 체크해도 같은 구조로 저장할 수 있다.
    
- 사용자가 한 번에 제출해도 날짜별 기록이 남는다.
    
- 캘린더 화면을 만들기 쉽다.
    
- 월간 통계를 날짜 기준으로 정확하게 계산할 수 있다.
    
- 주간 제출 여부와 벌금 계산 기준을 분리해서 관리할 수 있다.
    

---

## 벌금 규칙 API 설계

벌금 규칙은 코드에 하드코딩하지 않고 데이터로 관리하도록 설계했다.

현재 운영 규칙은 큐티, 기도, 말씀읽기, 토요 지각처럼 항목마다 계산 방식이 다르다.

큐티, 기도, 말씀읽기는 부족 횟수 기준으로 계산한다.

반면 지각은 지각 시간이 있을 때 기본 금액과 분당 금액을 함께 계산해야 한다.

그래서 벌금 규칙에는 규칙 유형과 계산 방식을 분리해서 두었다.

|규칙 유형|계산 방식|
|---|---|
|큐티|부족 횟수 기준|
|기도|부족 횟수 기준|
|말씀읽기|부족 횟수 기준|
|토요 지각|지각 시간 기준|

이 구조를 사용하면 캠퍼스마다 벌금 금액이나 기준 횟수가 달라져도 코드 수정 없이 데이터 설정으로 대응할 수 있다.

예를 들어 큐티 기준 횟수나 금액이 변경되어도 벌금 규칙만 수정하면 된다.

## 청구 항목 API 설계

처음에는 벌금과 커피비를 따로 관리하는 구조도 생각했다.

하지만 실제 사용자 입장에서는 둘 다 “내가 내야 할 돈”이다.

벌금인지 커피비인지는 발생 원인일 뿐이고, 납부 흐름은 같다.

그래서 벌금과 커피비를 하나의 청구 항목으로 통합했다.

|청구 종류|설명|
|---|---|
|벌금|경건생활 미달 또는 지각으로 발생|
|커피비|커피 투표 응답으로 발생|

청구 항목은 자신이 어디서 발생했는지 추적할 수 있어야 한다.

예를 들어 벌금은 주간 경건생활 기록에서 발생하고, 커피비는 투표 응답에서 발생한다.

그래서 청구 항목에는 발생 원인을 구분할 수 있는 값을 둔다.

사용자 화면에서는 다음과 같은 정보를 보여줄 수 있도록 API 응답을 구성했다.

```json
{
  "summary": {
    "totalAmount": 8500,
    "unpaidAmount": 6500,
    "paidAmount": 2000
  },
  "items": [
    {
      "title": "경건생활 벌금",
      "reason": "필수 횟수보다 적게 제출",
      "amount": 1000,
      "status": "UNPAID",
      "account": {
        "bankName": "은행명",
        "accountNumber": "0000-00-000000",
        "accountHolder": "예금주"
      }
    }
  ]
}
```

실제 계좌 정보와 사용자 정보는 블로그에 공개하지 않는다.

위 값은 API 응답 형태를 설명하기 위한 예시다.

---

## 벌금은 왜 한 줄로 청구하는가

벌금을 저장하는 방식도 고민이 있었다.

처음에는 큐티 벌금, 기도 벌금, 말씀 벌금, 지각 벌금을 각각 별도 청구 항목으로 만들 수 있다고 생각했다.

예를 들면 다음과 같다.

|청구 항목|금액|
|---|---|
|큐티 부족 벌금|1000원|
|기도 부족 벌금|500원|
|지각 벌금|1500원|

하지만 실제 사용자는 벌금을 항목별로 따로 납부하기보다, 한 주의 벌금을 한 번에 납부하는 방식이 더 자연스럽다.

그래서 MVP에서는 세부 항목을 계산한 뒤, 실제 청구 항목은 주차별로 한 줄만 생성하는 방향이 더 단순하다고 판단했다.

예를 들어 큐티 부족과 지각이 같이 있다면 하나의 청구 항목으로 만들고, 상세 사유는 reason에 요약해서 남긴다.

나중에 벌금 상세 화면이 필요해지면 별도의 상세 테이블을 추가할 수 있지만, 초기 버전에서는 한 줄 청구 구조가 운영과 구현 모두에 더 적합하다고 판단했다.

---

## 계좌 정보를 스냅샷으로 저장한 이유

청구 항목에는 납부 계좌 정보도 함께 들어간다.

처음에는 청구 항목에서 납부 계좌 ID만 참조하면 충분하다고 생각했다.

하지만 계좌는 나중에 바뀔 수 있다.

예를 들어 6월에는 A계좌로 벌금을 걷다가, 7월부터 B계좌로 변경될 수 있다.

이때 6월에 생성된 청구 항목까지 B계좌로 보이면 과거 기록이 바뀌는 문제가 생긴다.

그래서 청구 항목이 생성될 때 당시 계좌 정보를 스냅샷으로 저장하도록 했다.

이렇게 하면 계좌가 변경되어도 과거 청구 항목은 당시의 납부 정보를 유지할 수 있다.

---

## 사용자의 납부 완료 처리

실제 운영에서는 사용자가 계좌로 돈을 보낸 뒤, 본인의 청구 항목을 납부 완료 처리할 수 있어야 한다.

다만 이 기능은 권한 검증이 중요하다.

사용자는 다음 조건을 만족하는 경우에만 납부 완료 처리를 할 수 있어야 한다.

- 해당 캠퍼스에 속한 사용자여야 한다.
    
- 본인에게 청구된 항목이어야 한다.
    
- 아직 미납 상태인 항목이어야 한다.
    

다른 사람의 청구 항목을 수정하거나, 이미 취소 또는 면제된 항목을 임의로 바꾸면 안 된다.

그래서 사용자가 직접 납부 완료 처리할 수 있는 기능은 두되, 본인 소유의 청구 항목에만 적용되도록 설계했다.

관리자는 운영 중 실수나 예외 상황을 처리하기 위해 별도의 상태 정정 기능을 가진다.

다만 실제 상태 변경 API의 상세 경로나 검증 조건은 운영 보안을 위해 블로그에는 공개하지 않는다.

## 투표 API 설계

FaithLog의 투표는 단순한 설문 기능이 아니다.

수요예배 참석 투표, 토요 목자모임 참석 투표, 커피 주문 투표처럼 운영과 직접 연결되는 기능이다.

그래서 투표를 공통 구조로 설계했다.

|투표 유형|설명|
|---|---|
|수요예배 참석|참석 여부 확인|
|토요 목자모임|참석 여부 확인|
|커피 주문|메뉴 선택 및 금액 생성|
|커스텀|기타 운영용 투표|

수요예배와 토요 목자모임 투표는 금액이 발생하지 않는다.

반면 커피 주문 투표는 사용자가 선택한 메뉴 가격으로 청구 항목이 생성된다.

그래서 투표에는 청구 생성 방식이라는 개념을 두었다.

|청구 생성 방식|설명|
|---|---|
|없음|투표 응답만 저장|
|선택지 가격 기준|선택한 옵션의 금액으로 청구 생성|

커피 투표에서는 선택지마다 메뉴명과 금액을 둘 수 있다.

사용자가 특정 메뉴를 선택하면 해당 가격으로 청구 항목이 생성된다.

이렇게 하면 커피 주문과 커피비 정산을 하나의 흐름으로 처리할 수 있다.

---

## 투표 템플릿을 둔 이유

수요예배 참석 투표와 토요 목자모임 투표는 매주 반복된다.

매번 같은 제목과 선택지를 새로 입력하는 것은 비효율적이다.

그래서 반복 투표를 위한 템플릿을 두었다.

템플릿에는 투표 제목, 투표 유형, 시작 요일, 시작 시간, 마감 요일, 마감 시간, 기본 선택지를 저장한다.

이후 실제 투표는 템플릿을 기반으로 생성할 수 있다.

이 구조를 사용하면 반복되는 투표를 자동 생성하거나, 관리자가 빠르게 생성할 수 있다.

---

## 투표 미참여자 조회를 분리한 이유

투표 API를 설계하면서 결과 조회와 미참여자 조회를 분리했다.

처음에는 투표 결과 응답 안에 미참여자 목록까지 포함하면 된다고 생각했다.

하지만 실제 관리 화면을 생각해보니 두 기능의 목적이 달랐다.

투표 결과 조회는 선택지별 응답 수를 확인하는 기능이다.

반면 미참여자 조회는 아직 응답하지 않은 사람을 찾아 알림을 보내기 위한 기능이다.

그래서 두 기능을 분리했다.

|기능|목적|
|---|---|
|투표 결과 조회|선택지별 응답 수 확인|
|미참여자 조회|알림 대상자 확인|

이렇게 분리하면 관리자는 투표 결과를 확인하고, 필요한 경우 미참여자에게만 알림을 보낼 수 있다.

---

## 알림 API 설계

알림은 FCM 기반으로 설계했다.

사용자는 기기에서 발급받은 FCM 토큰을 서버에 등록한다.

서버는 특정 상황이 발생했을 때 대상자를 조회하고 알림을 발송한다.

대표적인 알림 상황은 다음과 같다.

- 경건생활 제출 안내
    
- 경건생활 미제출 알림
    
- 수요예배 투표 시작 알림
    
- 수요예배 투표 미참여 알림
    
- 토요 목자모임 투표 시작 알림
    
- 토요 목자모임 투표 미참여 알림
    
- 커피 투표 시작 알림
    
- 커피 투표 미참여 알림
    
- 미납 금액 알림
    
- 관리자 커스텀 알림
    

알림은 단순히 보내고 끝나는 기능이 아니다.

실제 운영에서는 알림 전송이 실패할 수도 있다.

그래서 사용자별 알림 발송 결과를 로그로 남기도록 설계했다.

누구에게 알림이 갔는지, 실패한 경우 이유가 무엇인지 확인할 수 있어야 운영 중 문제를 파악할 수 있기 때문이다.

---

## 월간 통계 API 설계

사용자 입장에서는 이번 달 자신의 상태를 한눈에 볼 수 있어야 한다.

그래서 월간 통계 API를 설계했다.

월간 통계에는 다음 정보를 포함한다.

- 이번 달 총 청구 금액
    
- 이번 달 납부 완료 금액
    
- 이번 달 미납 금액
    
- 전체 누적 납부 금액
    
- 큐티 체크 횟수
    
- 기도 체크 횟수
    
- 말씀읽기 체크 횟수
    
- 토요 지각 시간
    

여기서 경건생활 횟수는 주간 기록만으로 계산하지 않고 날짜별 체크 데이터를 기준으로 계산한다.

이유는 월이 걸쳐 있는 주차가 있기 때문이다.

예를 들어 한 주가 5월 말과 6월 초에 걸쳐 있다면, 주간 기록만 기준으로 계산할 경우 월간 통계가 부정확해질 수 있다.

날짜별 체크 데이터를 기준으로 계산하면 월별 통계를 더 정확하게 만들 수 있다.

---

## 정리

이번 API 설계에서는 단순히 기능별 CRUD를 나열하는 것이 아니라, 실제 운영 흐름을 기준으로 구조를 잡는 데 집중했다.

특히 다음과 같은 점을 중심으로 설계했다.

- 캠퍼스 중심의 API 구조
    
- 서비스 전체 권한과 캠퍼스 내부 역할 분리
    
- 초대코드 기반 캠퍼스 가입
    
- 하루 체크와 주간 제출을 모두 지원하는 경건생활 API
    
- 캠퍼스별 벌금 규칙 관리
    
- 벌금과 커피비를 통합한 청구 항목 구조
    
- 주차별 한 줄 벌금 청구 구조
    
- 계좌 스냅샷 저장
    
- 사용자 직접 납부 완료 처리
    
- 투표 미참여자 조회와 알림 발송 흐름
    
- 월간 경건생활 및 납부 통계 제공
    

ERD를 설계할 때는 데이터를 어떻게 저장할지만 고민했다면, API를 설계하면서는 실제 사용자가 어떤 흐름으로 서비스를 사용할지 더 구체적으로 생각하게 되었다.

API 설계는 단순히 엔드포인트를 나열하는 작업이 아니라, 운영 흐름과 데이터 구조를 연결하는 작업이라는 것을 느꼈다.

다음 단계에서는 이 설계를 바탕으로 도메인별 패키지 구조를 정리하고, 인증/인가부터 구현을 시작할 예정이다.
