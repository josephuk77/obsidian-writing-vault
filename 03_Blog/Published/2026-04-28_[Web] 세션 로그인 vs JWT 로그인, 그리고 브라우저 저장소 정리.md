# [Web] 세션 로그인 vs JWT 로그인, 그리고 브라우저 저장소 정리

로그인을 공부하다 보면 세션, JWT, 쿠키, 세션 스토리지, 로컬 스토리지 같은 단어가 계속 나온다.

처음에는 이 개념들이 각각 따로 있는 것처럼 느껴졌는데, 공부해보니까 전부 로그인 상태를 어떻게 유지할 것인지와 관련이 있었다.

이번 글에서는 세션 로그인과 JWT 로그인의 차이를 먼저 정리하고, 그다음 쿠키와 브라우저 저장소가 로그인에서 어떤 역할을 하는지 정리해보려고 한다.

그리고 마지막으로 React에서도 세션 로그인을 사용할 수 있는지까지 정리해보겠다.

---

## 로그인을 왜 유지해야 할까

로그인은 단순히 아이디와 비밀번호를 확인하는 과정만을 의미하지 않는다.

로그인에서 중요한 건 로그인 이후다.

사용자가 로그인에 성공한 뒤에 마이페이지에 들어가거나, 게시글을 작성하거나, 장바구니를 조회한다고 생각해보자.

이때 서버는 이런 걸 알아야 한다.

```text
이 요청을 보낸 사용자가 누구인가?
```

하지만 HTTP는 기본적으로 상태를 기억하지 않는다.

예를 들어 사용자가 한 번 로그인을 했다고 해서 서버가 다음 요청에서도 자동으로 그 사용자를 기억하는 것은 아니다.

그래서 로그인 이후에는 사용자를 계속 식별할 수 있는 방법이 필요하다.

그 방법 중 대표적인 것이 세션 로그인과 JWT 로그인이다.

---

## 세션 로그인

세션 로그인은 서버가 로그인 상태를 기억하는 방식이다.

사용자가 로그인을 하면 서버는 사용자의 로그인 정보를 확인한다.  
로그인이 성공하면 서버는 세션을 만들고, 그 세션을 구분할 수 있는 세션 ID를 브라우저에게 보내준다.

브라우저는 이 세션 ID를 보통 쿠키에 저장한다.

![](https://velog.velcdn.com/images/josephuk77/post/a28abd7b-776f-45db-a6cd-4b0142457d17/image.png)


세션 로그인 흐름은 대략 이렇게 된다.

```text
사용자가 로그인 요청을 보낸다
서버가 아이디와 비밀번호를 확인한다
로그인이 성공하면 서버가 세션을 생성한다
서버가 세션 ID를 쿠키로 브라우저에 전달한다
브라우저는 이후 요청마다 쿠키를 자동으로 함께 보낸다
서버는 쿠키에 있는 세션 ID를 보고 사용자를 확인한다
```

여기서 중요한 점은 로그인 정보를 서버가 가지고 있다는 것이다.

브라우저는 사용자의 실제 정보를 들고 있는 게 아니라, 서버에 저장된 세션을 찾을 수 있는 세션 ID만 가지고 있다.

그래서 세션 로그인은 이렇게 정리할 수 있다.

```text
세션 로그인은 서버가 로그인 상태를 기억하는 방식이다.
```

---

## JWT 로그인

JWT 로그인은 세션 로그인과 다르게 서버가 로그인 상태를 직접 저장하지 않는다.

사용자가 로그인을 하면 서버는 사용자의 정보를 확인한 뒤 JWT라는 토큰을 발급한다.

클라이언트는 이 토큰을 저장하고 있다가, 이후 API 요청을 보낼 때마다 토큰을 함께 보낸다.

![JWT 로그인 흐름](https://velog.velcdn.com/images/josephuk77/post/6effc6e1-c51d-4f12-a13d-ab43b4d8a936/image.png)


JWT 로그인 흐름은 대략 이렇게 된다.

```text
사용자가 로그인 요청을 보낸다
서버가 아이디와 비밀번호를 확인한다
로그인이 성공하면 서버가 JWT를 발급한다
클라이언트가 JWT를 저장한다
이후 요청마다 JWT를 Authorization 헤더에 담아서 보낸다
서버는 JWT를 검증해서 사용자를 확인한다
```

세션 로그인과 가장 큰 차이는 서버가 로그인 상태를 저장하지 않는다는 점이다.

JWT 안에는 사용자를 식별할 수 있는 정보가 들어 있고, 서버는 요청이 올 때마다 이 토큰이 유효한지만 검증한다.

그래서 JWT 로그인은 이렇게 정리할 수 있다.

```text
JWT 로그인은 클라이언트가 토큰을 들고 다니는 방식이다.
```

---

## 세션과 JWT의 핵심 차이

세션과 JWT의 차이는 한 문장으로 정리할 수 있다.

```text
세션은 서버가 기억하고,
JWT는 클라이언트가 들고 다닌다.
```

조금 더 비교해보면 이렇게 볼 수 있다.

| 구분 | 세션 로그인 | JWT 로그인 |
| --- | --- | --- |
| 로그인 상태 저장 위치 | 서버 | 클라이언트 |
| 클라이언트가 들고 있는 것 | 세션 ID | JWT |
| 주로 사용하는 저장 방식 | 쿠키 | 로컬 스토리지, 세션 스토리지, 쿠키 |
| 서버의 역할 | 세션 저장 및 조회 | 토큰 검증 |
| 장점 | 구현이 비교적 단순하고 안정적 | 서버 확장에 유리함 |
| 단점 | 서버가 세션을 관리해야 함 | 토큰 관리와 보안 처리가 중요함 |

처음에는 JWT가 더 최신 기술이고 무조건 좋은 방식이라고 생각했는데, 꼭 그런 것은 아니었다.

서비스 구조나 목적에 따라 세션이 더 적합할 수도 있고, JWT가 더 적합할 수도 있다.

---

## 쿠키

쿠키는 브라우저에 데이터를 저장하는 방법 중 하나다.

쿠키의 가장 큰 특징은 서버로 요청을 보낼 때 자동으로 함께 전송된다는 것이다.

예를 들어 세션 로그인에서는 서버가 세션 ID를 쿠키에 담아서 브라우저로 보내준다.

그러면 브라우저는 다음 요청부터 그 쿠키를 자동으로 서버에 함께 보낸다.

```text
브라우저 요청
→ 쿠키가 자동으로 같이 전송됨
→ 서버가 쿠키 안의 세션 ID 확인
→ 로그인한 사용자 확인
```

그래서 세션 로그인과 쿠키는 자주 같이 나온다.

다만 쿠키는 단순히 세션 로그인에서만 쓰이는 것은 아니다.  
JWT도 쿠키에 저장할 수 있다.

즉, 쿠키는 로그인 방식이 아니라 브라우저 저장소 중 하나라고 보는 게 맞다.

![쿠키와 서버 요청](https://velog.velcdn.com/images/josephuk77/post/22ed0daa-3525-4a04-99a3-96f1b60c4773/image.png)

---

## 세션 스토리지

세션 스토리지는 브라우저의 저장 공간 중 하나다.

이름 때문에 세션 로그인과 헷갈릴 수 있는데, 둘은 같은 개념이 아니다.

세션 로그인에서 말하는 세션은 서버가 관리하는 로그인 상태를 의미한다.

반면 세션 스토리지는 브라우저가 제공하는 저장소다.

세션 스토리지의 특징은 탭 단위로 데이터가 유지된다는 것이다.

```text
브라우저 탭을 유지하는 동안 데이터가 남아 있음
탭을 닫으면 데이터가 사라짐
서버로 자동 전송되지 않음
```

세션 스토리지에 저장한 값은 서버로 자동 전송되지 않는다.

그래서 JWT를 세션 스토리지에 저장했다면, API 요청을 보낼 때 직접 꺼내서 Authorization 헤더에 넣어줘야 한다.

```javascript
const token = sessionStorage.getItem("accessToken");

fetch("/api/user", {
  headers: {
    Authorization: `Bearer ${token}`,
  },
});
```

---

## 로컬 스토리지

로컬 스토리지도 브라우저의 저장 공간이다.

세션 스토리지와 다른 점은 브라우저를 껐다 켜도 데이터가 유지된다는 것이다.

```text
브라우저를 종료해도 데이터가 남아 있음
직접 삭제하기 전까지 유지됨
서버로 자동 전송되지 않음
```

JWT 로그인 예제를 보면 토큰을 로컬 스토리지에 저장하는 경우가 많다.

```javascript
localStorage.setItem("accessToken", token);
```

그리고 요청을 보낼 때는 다시 꺼내서 사용한다.

```javascript
const token = localStorage.getItem("accessToken");

fetch("/api/user", {
  headers: {
    Authorization: `Bearer ${token}`,
  },
});
```

하지만 로컬 스토리지는 보안적으로 조심해야 한다.

자바스크립트로 접근할 수 있기 때문에 XSS 공격에 취약할 수 있다.

그래서 실제 서비스에서는 토큰을 어디에 저장할지도 신중하게 결정해야 한다.

---

## 쿠키, 세션 스토리지, 로컬 스토리지 비교

쿠키, 세션 스토리지, 로컬 스토리지는 모두 브라우저에 데이터를 저장할 수 있다는 공통점이 있다.

하지만 동작 방식은 다르다.

![브라우저 저장소 비교](https://velog.velcdn.com/images/josephuk77/post/f387c307-da3c-42ab-8a6b-043d513bf0f7/image.png)


| 구분 | 쿠키 | 세션 스토리지 | 로컬 스토리지 |
| --- | --- | --- | --- |
| 저장 위치 | 브라우저 | 브라우저 | 브라우저 |
| 유지 기간 | 설정에 따라 다름 | 탭을 닫을 때까지 | 직접 삭제하기 전까지 |
| 서버 자동 전송 | 자동 전송됨 | 자동 전송 안 됨 | 자동 전송 안 됨 |
| 주 사용 예시 | 세션 ID 저장 | 임시 데이터 저장 | 장기 저장 데이터 |
| 로그인과의 관계 | 세션 로그인에서 자주 사용 | JWT 저장에 사용 가능 | JWT 저장에 사용 가능 |

여기서 중요한 건 이거다.

```text
쿠키, 세션 스토리지, 로컬 스토리지는 로그인 방식이 아니라 저장 위치다.
```

즉, 세션 로그인인지 JWT 로그인인지와  
쿠키를 쓰는지 스토리지를 쓰는지는 구분해서 이해해야 한다.

## React에서도 세션 로그인을 사용할 수 있을까

React를 공부하다 보면 JWT 로그인 예제를 많이 보게 된다.

그래서 React에서는 세션 로그인을 못 쓰는 것처럼 느껴질 수 있다.

하지만 React에서도 세션 로그인을 사용할 수 있다.

다만 정확히 말하면 React가 세션 로그인을 직접 하는 것은 아니다.

```text
세션 관리는 서버가 하고,
React는 요청을 보내고 응답을 받는 역할을 한다.
```

예를 들어 Spring Boot와 React를 같이 사용한다고 하면 흐름은 이렇게 된다.

```text
React에서 로그인 요청을 보낸다
Spring Boot 서버가 로그인 정보를 확인한다
서버가 세션을 생성한다
서버가 세션 ID를 쿠키로 내려준다
브라우저는 쿠키를 저장한다
React가 API 요청을 보내면 브라우저가 쿠키를 자동으로 함께 보낸다
서버는 세션을 확인해서 로그인 여부를 판단한다
```

React 입장에서는 그냥 로그인 API를 호출하고, 이후 필요한 API를 호출하는 것이다.

실제로 로그인 상태를 기억하는 건 서버다.

---

## React에서 JWT 로그인을 사용한다면

JWT 로그인을 사용할 때도 React가 모든 인증을 처리하는 것은 아니다.

토큰을 발급하고 검증하는 건 서버의 역할이다.

React는 발급받은 토큰을 저장하고, 요청을 보낼 때 토큰을 함께 보내는 역할을 한다.

```text
React에서 로그인 요청을 보낸다
서버가 JWT를 발급한다
React가 JWT를 저장한다
React가 API 요청을 보낼 때 JWT를 헤더에 담아서 보낸다
서버가 JWT를 검증한다
```

예를 들면 이런 식이다.

```javascript
fetch("/api/user", {
  headers: {
    Authorization: `Bearer ${accessToken}`,
  },
});
```

즉, React는 인증 방식을 결정하는 주체라기보다 서버에서 정한 인증 방식을 사용하는 쪽에 가깝다.

---

## 세션 로그인은 언제 좋을까

세션 로그인은 비교적 단순한 서비스에 잘 어울린다.

예를 들어 일반적인 웹 서비스, 관리자 페이지, 사내 시스템, 규모가 크지 않은 프로젝트에서는 세션 로그인이 충분히 좋은 선택이 될 수 있다.

세션 로그인의 장점은 서버가 로그인 상태를 관리하기 때문에 제어하기 쉽다는 점이다.

예를 들어 로그아웃을 하면 서버에서 세션을 삭제하면 된다.

```text
서버에서 세션 삭제
→ 해당 세션 ID는 더 이상 유효하지 않음
→ 로그아웃 처리 완료
```

이런 방식은 이해하기도 쉽고 구현도 비교적 단순하다.

하지만 서버가 세션을 저장하고 있어야 하기 때문에 서버가 여러 대로 늘어나면 세션을 어떻게 공유할지 고민해야 한다.

---

## JWT 로그인은 언제 좋을까

JWT 로그인은 서버가 로그인 상태를 저장하지 않아도 된다는 장점이 있다.

그래서 여러 서버가 있는 구조나, 모바일 앱과 웹이 함께 사용하는 구조에서 자주 사용된다.

서버는 토큰이 유효한지만 검증하면 되기 때문에 확장성 측면에서 장점이 있다.

하지만 JWT는 한 번 발급된 토큰을 중간에 무효화하기 어렵다는 문제가 있다.

예를 들어 사용자가 로그아웃을 해도, 이미 발급된 토큰 자체는 만료 시간 전까지 유효할 수 있다.

그래서 실제로는 access token, refresh token, 토큰 만료 시간, 재발급 처리 같은 것들을 함께 고민해야 한다.

JWT는 편해 보이지만 제대로 구현하려면 생각할 게 많다.

## 내가 정리한 선택 기준

공부하면서 정리한 기준은 이렇다.

```text
간단한 웹 서비스라면 세션 로그인이 더 단순하고 좋을 수 있다.
확장성이나 여러 클라이언트를 고려해야 한다면 JWT 로그인이 더 적합할 수 있다.
```

처음에는 JWT가 더 좋은 방식이라고 생각했다.

하지만 정리해보니까 세션과 JWT는 누가 더 좋다기보다 상황에 따라 선택하는 방식이었다.

특히 처음 로그인을 공부하거나, Spring Boot와 React로 일반적인 웹 프로젝트를 만든다면 세션 로그인부터 이해하는 것도 좋다고 생각한다.

그리고 JWT를 사용할 때는 단순히 토큰을 발급하고 저장하는 것에서 끝나는 게 아니라, 토큰을 어디에 저장할지, 만료 처리는 어떻게 할지, 보안 문제는 어떻게 막을지도 같이 생각해야 한다.

## 마무리

이번에 세션 로그인, JWT 로그인, 쿠키, 세션 스토리지, 로컬 스토리지를 정리하면서 가장 헷갈렸던 건 비슷한 단어들이 서로 다른 개념을 가리킨다는 점이었다.

특히 세션 로그인과 세션 스토리지는 이름은 비슷하지만 완전히 다른 개념이다.

정리하면 이렇게 볼 수 있다.

```text
세션 로그인은 서버가 로그인 상태를 기억하는 방식이다.
JWT 로그인은 클라이언트가 토큰을 들고 다니는 방식이다.
쿠키, 세션 스토리지, 로컬 스토리지는 브라우저 저장소다.
React는 서버가 정한 인증 방식을 사용한다.
```

결국 중요한 건 무조건 JWT를 쓰는 것도 아니고, 무조건 세션을 쓰는 것도 아니다.

내가 만드는 서비스의 구조에 맞게 선택하는 게 중요하다.
