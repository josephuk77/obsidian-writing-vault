# [Network] OSI 7계층과 실제 데이터 전송 흐름 정리

네트워크를 공부할 때 OSI 7계층을 단순히 외우기만 하면 금방 헷갈린다.  
정말 중요한 것은 각 계층에서 **무슨 역할을 하는지**, **어떤 장치가 관련되는지**, 그리고 **데이터가 어떤 형태로 바뀌는지**를 함께 이해하는 것이다.

이 글에서는 OSI 7계층 설명부터 실제 데이터 송수신 과정, ARP, NAT, MAC 주소 변화까지 하나의 흐름으로 정리해보겠다.

---

## 1. OSI 7계층이란?

OSI 7계층은 네트워크 통신 과정을 7단계로 나누어 설명한 모델이다.  
데이터는 송신 측에서 위에서 아래로 내려가며 전송 가능한 형태로 바뀌고, 수신 측에서는 아래에서 위로 올라가며 원래 데이터로 복원된다.

즉,

- 송신 측: 캡슐화(Encapsulation)
- 수신 측: 디캡슐화(Decapsulation)

과정을 거친다고 볼 수 있다.

---

## 2. OSI 7계층 + 역할 + 장치 정리

| 계층 | 이름 | 역할 | 대표 프로토콜 | 관련 장치 |
|------|------|------|--------------|----------|
| 7계층 | Application | 사용자와 직접 만나는 계층 | HTTP, HTTPS, SMTP, FTP | PC, 스마트폰, 서버 |
| 6계층 | Presentation | 데이터 변환, 암호화, 압축 | SSL/TLS, JPEG, MPEG | PC, 서버 |
| 5계층 | Session | 연결 시작, 유지, 종료 | 세션 관리 프로토콜 | PC, 서버 |
| 4계층 | Transport | 데이터 분할, 신뢰성 보장, 포트 관리 | TCP, UDP | PC, 서버 |
| 3계층 | Network | IP 주소 기반 경로 결정 | IP, ICMP | 라우터 |
| 2계층 | Data Link | MAC 주소 기반 전달 | Ethernet, ARP | 스위치, NIC |
| 1계층 | Physical | 전기/광 신호로 실제 전송 | 케이블 규격, 전기 신호 | 허브, 케이블, 리피터 |

---

## 3. 외우기 좋은 방식

### 3-1. 계층 이름 외우기

응표세전네데물

- 응: Application
- 표: Presentation
- 세: Session
- 전: Transport
- 네: Network
- 데: Data Link
- 물: Physical

### 3-2. 데이터 형태로 외우기

데이터 → 세그먼트 → 패킷 → 프레임 → 비트

- Application ~ Session: 데이터
- Transport: 세그먼트
- Network: 패킷
- Data Link: 프레임
- Physical: 비트

### 3-3. 장치 흐름으로 외우기

PC → 스위치 → 라우터 → 인터넷 → 서버

- PC: 상위 계층 처리
- 스위치: MAC 주소 기반 전달
- 라우터: IP 주소 기반 전달
- 케이블: 신호 전달

---

## 4. 캡슐화란 무엇인가?

송신 측에서는 데이터가 아래 계층으로 내려가면서 각 계층의 헤더가 붙는다.  
이 과정을 캡슐화라고 한다.

예를 들어 "안녕"이라는 메시지를 보낸다고 하면, 처음에는 단순한 문자열이지만 아래로 내려가면서 다음처럼 바뀐다.

데이터 → TCP 정보가 붙은 세그먼트 → IP 정보가 붙은 패킷 → MAC 정보가 붙은 프레임 → 비트 신호

즉, **내용 자체가 바뀌는 것이 아니라 전송에 필요한 정보가 덧붙여지는 것**이다.

---

## 5. 전체 시나리오

이제 실제 상황을 가정해보자.

- 내 컴퓨터의 IP: 192.168.0.2
- 공유기 IP: 192.168.0.1
- 공유기의 공인 IP: 1.2.3.4
- 목적지 서버 IP: 8.8.8.8

내 컴퓨터에서 서버로 `"안녕"`이라는 데이터를 보내는 과정을 단계별로 살펴보겠다.

---

## 6. 송신 과정: 데이터가 내려가며 변화하는 과정

---

## 6-1. 7계층 Application

### 설명
Application 계층은 사용자가 실제로 접하는 계층이다.  
브라우저, 메신저, 이메일 앱 같은 프로그램이 이 계층에서 동작한다.

사용자가 메시지를 작성하거나 웹사이트에 요청을 보내면, 여기서 네트워크 통신이 시작된다.

### 데이터 형태
```text
데이터:
"안녕"
```
이 단계에서는 아직 순수한 데이터만 존재한다.  
IP 주소도 없고, MAC 주소도 없고, 포트 번호도 없다.

---

## 6-2. 6계층 Presentation

### 설명
Presentation 계층은 데이터를 표현 형식에 맞게 바꿔준다.  
예를 들어 문자 인코딩, 암호화, 압축 같은 작업이 여기서 이루어진다.

웹에서 HTTPS를 사용할 때 암호화가 적용되는 것도 이 계층과 관련 있다.

### 데이터 형태
```text
데이터:
[암호화 또는 인코딩된 "안녕"]
```
이 단계에서는 사람이 바로 읽기 어려운 형태로 바뀔 수 있다.  
하지만 아직도 전송 주소 정보는 붙지 않았다.

---

## 6-3. 5계층 Session

### 설명
Session 계층은 통신을 시작하고 유지하고 종료하는 역할을 한다.  
쉽게 말해 "지금 이 둘이 연결된 상태인지", "대화가 유지되고 있는지"를 관리하는 계층이다.

### 데이터 형태
```text
데이터:
[세션정보 + 암호화된 데이터]
```
이 단계에서는 세션 식별 정보가 포함될 수 있다.  
아직도 네트워크 전송에 필요한 IP나 MAC은 없다.

---

## 6-4. 4계층 Transport

### 설명
Transport 계층은 데이터를 실제로 전송할 수 있도록 나누고, 신뢰성을 관리한다.  
대표적으로 TCP와 UDP가 있다.

- TCP: 신뢰성 보장, 순서 보장
- UDP: 빠르지만 신뢰성은 낮음

또한 이 계층에서는 **포트 번호**가 붙는다.  
포트는 "어떤 프로그램으로 보내야 하는지"를 구분하는 역할을 한다.

예를 들어 웹 서버는 보통 80번 포트나 443번 포트를 사용한다.

### 데이터 형태
```text
세그먼트:
[출발포트:5000 | 도착포트:80 | 세션정보 | 데이터]
```
여기까지 오면 "어떤 프로그램에서 어떤 프로그램으로 보내는지"는 정해졌지만,  
"어느 컴퓨터로 보낼지"는 아직 정해지지 않았다.

---

## 6-5. 3계층 Network

### 설명
Network 계층에서는 **IP 주소**가 붙는다.  
이 계층의 핵심은 "어디까지 가야 하는가"를 결정하는 것이다.

즉,

- 출발지 IP
- 목적지 IP

를 이용해서 패킷이 어느 네트워크를 거쳐야 하는지 판단한다.

라우터가 주로 이 계층에서 동작한다.

### 데이터 형태
```text
패킷:
[출발IP:192.168.0.2 | 도착IP:8.8.8.8 | 출발포트:5000 | 도착포트:80 | 데이터]
```
이제 패킷은 "내 컴퓨터에서 서버로 가야 한다"는 정보를 가지게 되었다.  
하지만 실제 같은 네트워크 안에서 데이터를 바로 보내려면 MAC 주소가 필요하다.

---

## 6-6. 2계층 Data Link

### 설명
Data Link 계층은 **같은 네트워크 안에서 누구에게 전달할지**를 결정하는 계층이다.  
여기서는 IP가 아니라 **MAC 주소**를 사용한다.

MAC 주소는 네트워크 카드(NIC)에 부여된 물리적 주소이다.  
이 계층에서는 패킷 앞뒤로 MAC 정보와 오류 검사용 정보가 붙어서 프레임이 된다.

그런데 내 컴퓨터는 목적지 서버의 MAC 주소를 직접 알지 못한다.  
외부 인터넷으로 나갈 때는 우선 **공유기 MAC 주소**를 알아야 한다.  
이때 사용하는 것이 ARP이다.

### ARP 설명
ARP(Address Resolution Protocol)는  
"이 IP 주소를 가진 장치의 MAC 주소가 무엇인가?"를 알아내는 프로토콜이다.

예를 들어 내 컴퓨터는 먼저 공유기에게 물어본다.
```text
ARP 요청:
"192.168.0.1의 MAC 주소는 무엇인가?"
```
공유기가 응답한다.
```text
ARP 응답:
192.168.0.1 → AA:BB:CC:DD:EE:FF
```
이제 내 컴퓨터는 공유기 MAC 주소를 알게 되었고, 프레임을 만들 수 있다.

### 데이터 형태
```text
프레임:
[출발MAC:내MAC | 도착MAC:공유기MAC | 출발IP:192.168.0.2 | 도착IP:8.8.8.8 | 출발포트:5000 | 도착포트:80 | 데이터 | 오류검사]
```
여기서 중요한 점은,  
**IP는 최종 목적지를 나타내고, MAC은 바로 다음 장치를 나타낸다**는 것이다.

---

## 6-7. 1계층 Physical

### 설명
Physical 계층은 실제로 케이블이나 무선 신호를 통해 데이터를 보내는 계층이다.  
이 계층에서는 0과 1의 비트가 전기 신호, 빛 신호, 전파 형태로 바뀌어 전송된다.

### 데이터 형태
```text
비트:
010101010101010101...
```
이제 데이터는 실제 신호가 되어 공유기로 전송된다.

---

## 7. 공유기에서 일어나는 일: NAT

### 설명
공유기는 라우터 역할을 하며 3계층 장치이다.  
내 컴퓨터가 보낸 패킷은 먼저 공유기에 도착한다.

그런데 내 컴퓨터의 IP인 `192.168.0.2`는 사설 IP라서 인터넷에서 그대로 사용할 수 없다.  
그래서 공유기는 NAT(Network Address Translation)를 수행한다.

즉, 출발지 IP를 공유기의 공인 IP로 바꾼다.

### 변경 전 데이터
```text
패킷:
[출발IP:192.168.0.2 | 도착IP:8.8.8.8 | 출발포트:5000 | 도착포트:80 | 데이터]
```
### 변경 후 데이터
```text
패킷:
[출발IP:1.2.3.4 | 도착IP:8.8.8.8 | 출발포트:5000 | 도착포트:80 | 데이터]
```
즉, 외부 인터넷에서는 이 요청이 내 컴퓨터가 아니라 **공유기에서 보낸 요청처럼 보이게 된다**.

---

## 8. 인터넷 구간에서의 이동

### 설명
공유기에서 나온 패킷은 여러 라우터를 거치면서 목적지 서버로 이동한다.

이때 중요한 특징이 있다.

- IP 주소는 최종 목적지를 나타내므로 계속 유지된다.
- MAC 주소는 "바로 다음 장치"를 나타내므로 라우터를 지날 때마다 계속 바뀐다.

즉, 인터넷 구간에서는 IP는 유지되고 MAC은 매 홉마다 새로 설정된다.

### 데이터 형태의 핵심 변화
```text
유지되는 정보:
출발IP:1.2.3.4
도착IP:8.8.8.8

변경되는 정보:
출발MAC, 도착MAC
```
---

## 9. 서버 도착

### 설명
패킷이 목적지 서버에 도착하면, 서버는 반대로 계층을 올라가며 데이터를 해석한다.  
즉, 프레임에서 MAC 정보를 제거하고, 패킷에서 IP 정보를 제거하고, 세그먼트에서 포트 정보를 확인해 최종적으로 데이터를 얻는다.

### 데이터 복원 과정
```text
프레임 → 패킷 → 세그먼트 → 데이터
```
### 최종 데이터
```text
"안녕"
```
결국 서버는 처음 사용자가 보낸 `"안녕"`을 받게 된다.

---

## 10. 응답이 돌아오는 과정

이제 서버가 응답을 보낸다고 가정해보자.  
흐름은 기본적으로 반대 방향이다.

---

## 10-1. 서버에서 응답 생성

### 설명
서버는 응답 데이터를 만든 뒤 다시 캡슐화해서 보낸다.

### 데이터 형태
```text
패킷:
[출발IP:8.8.8.8 | 도착IP:1.2.3.4 | 데이터]
```
여기서 도착 IP는 내 컴퓨터의 사설 IP가 아니라 공유기의 공인 IP이다.  
왜냐하면 서버는 NAT 이전의 내부 IP를 알지 못하기 때문이다.

---

## 10-2. 공유기에서 NAT 역변환

### 설명
공유기는 자신이 가지고 있는 NAT 테이블을 확인해  
`1.2.3.4:5000` 으로 온 응답이 사실은 `192.168.0.2:5000` 으로 가야 한다는 것을 안다.

그래서 도착 IP를 다시 내 컴퓨터의 사설 IP로 바꾼다.

### 데이터 형태
```text
변경 전:
[출발IP:8.8.8.8 | 도착IP:1.2.3.4 | 데이터]

변경 후:
[출발IP:8.8.8.8 | 도착IP:192.168.0.2 | 데이터]
```
---

## 10-3. Data Link 계층에서 다시 MAC 부여

### 설명
이제 공유기는 같은 네트워크 안에서 내 컴퓨터에게 보내야 하므로 내 MAC 주소를 알아야 한다.  
필요하다면 ARP를 통해 다시 MAC 주소를 확인한다.

그 후 프레임을 만들어 전송한다.

### 데이터 형태
```text
프레임:
[출발MAC:공유기MAC | 도착MAC:내MAC | 출발IP:8.8.8.8 | 도착IP:192.168.0.2 | 데이터]
```
---

## 10-4. 내 컴퓨터에서 디캡슐화

### 설명
내 컴퓨터는 Physical 계층에서 비트를 받고,  
Data Link 계층에서 MAC 정보를 확인한 뒤,  
Network 계층에서 IP 정보를 확인하고,  
Transport 계층에서 포트를 확인한 뒤,  
최종적으로 Application 계층에 데이터를 전달한다.

### 데이터 복원 과정
```text
비트 → 프레임 → 패킷 → 세그먼트 → 데이터
```
### 최종 결과
```text
"안녕"
```
결국 내가 보낸 데이터는 상대에게 전달되고, 상대의 응답도 다시 내게 돌아오게 된다.

---

## 11. 핵심 요약

### 11-1. 계층별 핵심 역할
- Application: 사용자 데이터 생성
- Presentation: 인코딩, 암호화, 압축
- Session: 연결 유지
- Transport: 포트, 신뢰성, 데이터 분할
- Network: IP 주소 기반 전달
- Data Link: MAC 주소 기반 전달
- Physical: 실제 신호 전송

### 11-2. 데이터 형태 변화
- 데이터
- 세그먼트
- 패킷
- 프레임
- 비트

### 11-3. 장치 역할
- PC/서버: 상위 계층 처리
- 스위치: MAC 주소 기반 전달
- 라우터: IP 주소 기반 전달
- 케이블/허브: 물리 신호 전달

### 11-4. 꼭 기억할 포인트
- IP는 최종 목적지를 위한 주소이다.
- MAC은 바로 다음 장치를 위한 주소이다.
- ARP는 IP를 이용해 MAC을 알아낸다.
- NAT는 사설 IP를 공인 IP로 바꾼다.
- 인터넷 구간에서는 MAC은 계속 바뀌고 IP는 유지된다.
- 수신 측에서는 헤더를 제거하며 원래 데이터를 복원한다.

## 12. 한 줄 정리

OSI 7계층에서 데이터는 상위 계층에서 하위 계층으로 내려가며 세그먼트, 패킷, 프레임, 비트 형태로 바뀌고, ARP로 MAC 주소를 알아내며, NAT로 사설 IP가 공인 IP로 변환된 뒤, 수신 측에서 다시 디캡슐화를 통해 원래 데이터로 복원된다.
