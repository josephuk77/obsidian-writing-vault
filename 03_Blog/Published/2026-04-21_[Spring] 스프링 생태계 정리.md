# [Spring] 스프링 생태계 정리

스프링을 공부하다 보면 처음에는 `Spring Boot`만 보이는데, 조금 지나면 생각보다 주변에 붙어 있는 기술들이 많다는 걸 알게 된다.

처음에는 단순히 서버를 띄우고 API를 만드는 프레임워크라고 생각했는데, 공부할수록 스프링은 하나의 프레임워크라기보다는 백엔드 개발을 위한 생태계에 가깝다는 생각이 들었다.

이번 글에서는 스프링 생태계가 어떤 식으로 구성되어 있는지, 그리고 MVC 요청이 실제로 어떤 흐름으로 실행되는지 정리해보려고 한다.

## 스프링 생태계 한눈에 보기

![](https://velog.velcdn.com/images/josephuk77/post/58c78043-1ac6-4efc-a45f-876f6d41167b/image.png)


스프링 생태계는 하나의 기술만 말하는 것이 아니다.

가장 기본이 되는 `Spring Framework`가 있고, 이 스프링을 더 쉽게 사용할 수 있게 해주는 `Spring Boot`가 있다. 그리고 웹 요청을 처리하는 `Spring MVC`, 데이터베이스 접근을 편하게 해주는 `Spring Data JPA`, 인증과 인가를 담당하는 `Spring Security` 같은 기술들이 함께 사용된다.

처음에는 이름이 많아서 헷갈릴 수 있지만, 결국 목적은 비슷하다.

개발자가 매번 반복적으로 작성해야 하는 설정이나 구조를 스프링이 대신 도와주고, 개발자는 비즈니스 로직에 더 집중할 수 있게 해주는 것이다.

## Spring Framework

가장 기본이 되는 것은 `Spring Framework`다.

스프링 프레임워크는 객체를 직접 생성하고 관리하는 방식에서 벗어나, 스프링 컨테이너가 객체를 관리하도록 도와준다.

여기서 핵심이 되는 개념이 `IoC`와 `DI`다.

`IoC`는 제어의 역전이라는 뜻인데, 객체의 생성과 의존관계 관리를 개발자가 직접 하는 것이 아니라 스프링에게 맡긴다는 의미다.

`DI`는 의존성 주입이다. 어떤 객체가 필요로 하는 다른 객체를 직접 생성하지 않고 외부에서 주입받는 방식이다.

```java
@Service
public class MemberService {

    private final MemberRepository memberRepository;

    public MemberService(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}
```

이런 식으로 작성하면 `MemberService`는 `MemberRepository`를 직접 만들지 않는다.

대신 스프링이 필요한 객체를 찾아서 넣어준다.

이 구조 덕분에 코드의 결합도가 낮아지고 테스트하기도 쉬워진다.

## Spring Boot

![](https://velog.velcdn.com/images/josephuk77/post/c415987e-5db5-48aa-b6ea-c6f61973d0ea/image.png)

스프링을 더 쉽게 사용할 수 있게 만들어주는 것이 `Spring Boot`다.

기존 스프링은 설정할 것이 많았다. 웹 서버 설정, 의존성 설정, 빈 등록, 환경 설정 등 신경 쓸 부분이 많았는데, 스프링 부트는 이런 부분을 많이 자동화해준다.

스프링 부트 애플리케이션은 보통 `main()` 메서드에서 시작한다.

```java
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

`SpringApplication.run()`이 실행되면 스프링 컨테이너가 만들어지고, 컴포넌트 스캔을 통해 `@Controller`, `@Service`, `@Repository` 같은 빈들이 등록된다.

그리고 필요한 설정들이 자동으로 적용되고, 내장 톰캣 같은 웹 서버가 실행된다.

그래서 별도의 서버를 직접 설치하지 않아도 애플리케이션을 바로 실행할 수 있다.

처음 스프링을 공부할 때는 보통 스프링 부트부터 시작한다. 그래서 스프링과 스프링 부트를 같은 것으로 생각하기 쉽지만, 정확히는 스프링 부트가 스프링 프레임워크를 더 편하게 사용할 수 있도록 도와주는 도구라고 보면 된다.

## Spring MVC

![](https://velog.velcdn.com/images/josephuk77/post/4100a5cd-ddde-4448-a31d-481415d28ea0/image.png)

웹 애플리케이션을 만들 때 자주 사용하는 것이 `Spring MVC`다.

Spring MVC는 클라이언트의 요청을 받고, 그 요청에 맞는 컨트롤러를 찾아 실행한 뒤 응답을 반환하는 구조를 제공한다.

```java
@RestController
@RequestMapping("/members")
public class MemberController {

    @GetMapping
    public String getMembers() {
        return "members";
    }
}
```

이런 식으로 컨트롤러를 작성하면 `/members`로 들어오는 요청을 처리할 수 있다.

요청 흐름을 간단히 보면 다음과 같다.

클라이언트가 요청을 보내면 가장 먼저 필터를 거친다. 여기에는 보안, CORS, 로그 처리 같은 작업이 들어갈 수 있다.

그 다음 요청은 `DispatcherServlet`으로 전달된다. `DispatcherServlet`은 스프링 MVC의 중심 역할을 하는 객체다. 모든 요청을 먼저 받아서 어떤 컨트롤러가 처리해야 하는지 판단한다.

이후 `HandlerMapping`이 요청 URL과 HTTP 메서드에 맞는 컨트롤러 메서드를 찾고, 컨트롤러가 실행된다.

컨트롤러는 보통 직접 복잡한 로직을 처리하지 않고 `Service`를 호출한다. 서비스는 비즈니스 로직을 처리하고, 데이터가 필요하면 `Repository`를 통해 데이터베이스에 접근한다.

마지막으로 처리 결과를 DTO나 JSON 형태로 클라이언트에게 응답한다.

## MVC 내부 흐름

![](https://velog.velcdn.com/images/josephuk77/post/ee6cdae8-d59f-4288-a95a-b73b033dc079/image.png)

Spring MVC를 조금 더 안쪽에서 보면 `DispatcherServlet`이 핵심이다.

`DispatcherServlet`은 요청을 받은 뒤 `HandlerMapping`을 통해 실행할 컨트롤러를 찾는다.

그리고 `HandlerAdapter`를 사용해서 실제 컨트롤러 메서드를 호출한다.

컨트롤러가 값을 반환하면, 일반적인 REST API에서는 그 값을 JSON으로 변환해서 응답한다.

반면 JSP나 Thymeleaf 같은 화면을 반환하는 방식에서는 `ViewResolver`가 뷰 이름에 맞는 화면 파일을 찾고, 그 결과를 HTML로 만들어 응답한다.

그래서 REST API를 만들 때는 `ViewResolver`를 직접 많이 의식하지 않지만, MVC 전체 구조를 이해할 때는 같이 알아두는 것이 좋다.

## Spring Data JPA

![](https://velog.velcdn.com/images/josephuk77/post/1bb5a828-6871-45ff-a336-b23ef0ae8689/image.png)

데이터베이스를 사용할 때는 `Spring Data JPA`를 많이 사용한다.

JPA는 자바 객체와 데이터베이스 테이블을 매핑해주는 기술이다. SQL을 직접 작성하는 대신 객체 중심으로 데이터를 다룰 수 있게 해준다.

Spring Data JPA는 JPA를 더 편하게 사용할 수 있도록 도와준다.

```java
public interface MemberRepository extends JpaRepository<Member, Long> {
    Optional<Member> findByEmail(String email);
}
```

이렇게 인터페이스만 작성해도 기본적인 CRUD 기능을 사용할 수 있다.

`save`, `findById`, `findAll`, `delete` 같은 메서드를 직접 구현하지 않아도 된다.

흐름을 보면 컨트롤러가 요청을 받고, 서비스에서 비즈니스 로직을 처리한다. 그 다음 레포지토리가 호출되고, 내부적으로는 `EntityManager`가 JPA 동작을 처리한다.

트랜잭션 안에서 엔티티가 조회되거나 변경되면 영속성 컨텍스트가 그 엔티티를 관리한다. 이후 트랜잭션이 커밋될 때 변경 내용을 감지하고 필요한 SQL을 데이터베이스에 전달한다.

처음에는 편해 보이지만, 내부 동작을 모르고 사용하면 오히려 문제가 생기기 쉽다. 특히 영속성 컨텍스트, 지연 로딩, N+1 문제 같은 개념은 꼭 이해하고 넘어가야 한다.

## Spring Security

![](https://velog.velcdn.com/images/josephuk77/post/96aba1b8-a4e6-4885-8c53-1510f33e5d72/image.png)

인증과 인가를 처리할 때 사용하는 것이 `Spring Security`다.

로그인, 로그아웃, 권한 확인, 비밀번호 암호화, JWT 인증 같은 기능을 구현할 때 자주 사용된다.

스프링 시큐리티는 컨트롤러보다 앞에서 동작한다.

클라이언트의 요청이 들어오면 먼저 `SecurityFilterChain`을 거친다. 이 과정에서 사용자가 누구인지 확인하는 인증이 진행되고, 해당 사용자가 이 API에 접근할 권한이 있는지 확인하는 인가가 진행된다.

인증에 성공하면 사용자 정보가 `SecurityContext`에 저장되고, 이후 컨트롤러에서 현재 로그인한 사용자 정보를 사용할 수 있다.

반대로 인증되지 않은 사용자라면 `401 Unauthorized`, 권한이 부족한 사용자라면 `403 Forbidden` 응답이 반환될 수 있다.

처음 접하면 설정이 복잡하게 느껴진다. 실제로 스프링 시큐리티는 러닝 커브가 있는 편이다.

하지만 인증과 보안은 대부분의 서비스에서 중요한 부분이기 때문에, 스프링으로 실무 프로젝트를 한다면 거의 반드시 만나게 되는 기술이다.

```java
.requestMatchers("/admin/**").hasRole("ADMIN")
.requestMatchers("/members/**").authenticated()
.anyRequest().permitAll()
```

이런 식으로 URL별 접근 권한을 설정할 수 있다.

## Spring Batch

대량의 데이터를 정해진 시간에 처리하거나 반복 작업을 자동화할 때는 `Spring Batch`를 사용한다.

예를 들면 다음과 같은 작업에 사용할 수 있다.

- 매일 새벽 정산 데이터 생성
- 오래된 로그 데이터 삭제
- 대량의 엑셀 데이터 업로드
- 외부 API 데이터를 주기적으로 수집
- 쿠폰 만료 처리

일반적인 API 요청은 사용자의 요청에 따라 바로 처리된다.

반면 배치 작업은 사용자가 직접 요청하지 않아도 특정 시간이나 조건에 따라 실행되는 경우가 많다.

Spring Batch는 이런 작업을 `Job`, `Step`, `Reader`, `Processor`, `Writer` 같은 구조로 나누어 처리한다.

처음에는 구조가 조금 낯설지만, 대량 데이터를 안정적으로 처리해야 하는 상황에서는 굉장히 유용하다.

## Spring Cloud

서비스 규모가 커지고 여러 개의 서버로 나뉘게 되면 `Spring Cloud`를 사용하게 될 수 있다.

Spring Cloud는 마이크로서비스 아키텍처를 구성할 때 필요한 여러 기능을 제공한다.

예를 들면 다음과 같은 기능들이 있다.

- 설정 중앙 관리
- 서비스 디스커버리
- API Gateway
- 분산 추적
- 장애 대응
- 서비스 간 통신

단일 애플리케이션에서는 크게 필요하지 않을 수 있지만, 서비스가 여러 개로 나뉘면 각각의 서비스가 서로를 어떻게 찾고 통신할지, 설정은 어떻게 관리할지, 장애가 났을 때 어떻게 대응할지 같은 문제가 생긴다.

Spring Cloud는 이런 문제들을 해결하기 위한 도구들을 제공한다.

## Spring WebFlux

일반적인 Spring MVC는 요청마다 스레드를 사용해서 처리하는 방식이다.

반면 `Spring WebFlux`는 비동기, 논블로킹 방식으로 요청을 처리한다.

많은 요청을 효율적으로 처리해야 하거나, 외부 API 호출이 많은 서비스에서 사용할 수 있다.

```java
@GetMapping("/members")
public Mono<String> getMember() {
    return Mono.just("member");
}
```

WebFlux에서는 `Mono`, `Flux` 같은 타입을 사용한다.

`Mono`는 데이터가 0개 또는 1개일 때 사용하고, `Flux`는 여러 개의 데이터를 다룰 때 사용한다.

다만 WebFlux는 무조건 Spring MVC보다 좋은 기술은 아니다. 구조가 다르고 학습 비용도 있기 때문에, 필요한 상황에서 사용하는 것이 좋다.

## Spring Actuator

애플리케이션을 운영하다 보면 현재 서버가 정상적으로 동작하는지 확인해야 한다.

이때 사용할 수 있는 것이 `Spring Boot Actuator`다.

Actuator를 사용하면 애플리케이션의 상태, 헬스 체크, 메트릭, 환경 정보 등을 확인할 수 있다.

예를 들어 다음과 같은 엔드포인트를 사용할 수 있다.

```text
/actuator/health
/actuator/metrics
```

운영 환경에서는 서버가 살아 있는지, 메모리는 얼마나 사용하는지, 요청은 얼마나 들어오는지 같은 정보를 확인하는 것이 중요하다.

Actuator는 이런 정보를 쉽게 확인할 수 있게 해준다.

## Spring Test

스프링은 테스트를 위한 기능도 잘 제공한다.

단위 테스트, 통합 테스트, MVC 테스트 등 다양한 방식으로 테스트할 수 있다.

```java
@SpringBootTest
class MemberServiceTest {

    @Test
    void test() {
        // test code
    }
}
```

또는 컨트롤러만 테스트하고 싶을 때는 `@WebMvcTest`를 사용할 수 있다.

테스트를 작성하면 코드가 제대로 동작하는지 확인할 수 있고, 나중에 코드를 수정했을 때 기존 기능이 망가지지 않았는지도 확인할 수 있다.

처음에는 테스트 코드를 작성하는 게 번거롭게 느껴지지만, 프로젝트가 커질수록 테스트의 중요성을 느끼게 된다.

## 스프링 생태계를 공부하면서 느낀 점

스프링 생태계는 처음 보면 범위가 너무 넓게 느껴진다.

Spring Framework, Spring Boot, Spring MVC, Spring Data JPA, Spring Security, Spring Batch, Spring Cloud 등 이름도 많고 각각의 역할도 다르다.

하지만 하나씩 보면 결국 목표는 비슷하다.

개발자가 비즈니스 로직에 더 집중할 수 있도록 반복적인 설정이나 복잡한 인프라 처리를 도와주는 것이다.

스프링을 처음 공부할 때 모든 기술을 한 번에 다 이해하려고 하면 어렵다.

개인적으로는 다음 순서로 공부하면 좋을 것 같다.

- Spring Framework의 핵심 개념 이해
- Spring Boot로 간단한 API 서버 만들기
- Spring MVC 구조 이해
- Spring Data JPA로 데이터베이스 연동
- Spring Security로 인증과 인가 구현
- 필요에 따라 Batch, Cloud, WebFlux 학습

처음부터 모든 생태계를 완벽하게 알 필요는 없다고 생각한다.

## 정리

스프링은 단순히 서버를 만드는 프레임워크가 아니라, 백엔드 개발 전반을 지원하는 거대한 생태계다.

Spring Framework는 핵심 기반을 제공하고, Spring Boot는 설정을 간단하게 만들어준다.

Spring MVC는 웹 요청을 처리하고, Spring Data JPA는 데이터베이스 접근을 편하게 만들어준다.

Spring Security는 인증과 인가를 담당하고, Spring Batch는 대량 작업을 처리할 때 사용된다.

Spring Cloud는 여러 서비스가 함께 동작하는 환경에서 도움을 주고, Actuator는 운영 중인 애플리케이션의 상태를 확인할 수 있게 해준다.
