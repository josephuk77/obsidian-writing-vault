# AI가 만든 코드 품질은 하네스로 좋아질 수 있을까

## 글을 쓰는 이유

요즘 AI 코딩 도구를 쓰면 코드를 정말 빠르게 만들 수 있다.

간단한 기능은 요구사항을 말로 설명하는 것만으로도 파일이 생기고, 테스트가 생기고, 화면까지 어느 정도 만들어진다.

처음에는 이 속도 자체가 가장 인상적이었다. 예전에는 직접 검색하고, 예제 코드를 보고, 파일 구조를 만들고, 하나씩 구현해야 했던 작업을 AI가 빠르게 처리해주기 때문이다.

그런데 조금 더 큰 작업을 맡기기 시작하면 다른 문제가 보인다.

AI가 코드를 빠르게 만드는 것은 맞다. 하지만 항상 내가 원하는 방향으로, 내가 생각한 품질 기준에 맞게, 프로젝트 전체 구조를 지키면서 만드는 것은 아니다.

처음 합의한 목표와 다른 방향으로 기능이 커지기도 하고, 테스트나 보안 검토가 빠지기도 한다. 작은 수정만 필요했는데 전체 구조를 다시 만지려고 하기도 한다. 한 세션에 너무 많은 맥락을 넣으면 중요한 결정이 흐려지고, 다음 세션에서는 또 같은 설명을 반복해야 한다.

그래서 요즘은 단순히 "AI에게 어떻게 더 잘 시킬까"보다 "AI가 일하는 구조를 어떻게 만들까"를 더 많이 고민하고 있다.

지금 만들고 싶은 것은 AI 개발 하네스다.

## 먼저 결론부터

아직 이 하네스가 실제로 코드 품질을 올린다고 검증된 것은 아니다.

지금 단계에서는 하나의 가설이다.

> AI에게 큰 작업을 통째로 맡기는 것보다, PM, 프레임워크 리드, 전문가, worker처럼 역할을 나누고, 작은 task packet 단위로 일하게 하고, 리뷰와 검증 게이트를 거치게 하면 코드 품질과 일관성이 올라갈 수 있지 않을까?

이 가설을 검증하기 위해 하네스를 만들고 있다.

여기서 하네스는 AI를 더 자유롭게 풀어놓는 장치가 아니다. 오히려 반대에 가깝다.

AI가 건드릴 수 있는 범위를 줄이고, 역할을 나누고, 구현 전에 결정을 기록하고, 구현 후에는 리뷰와 테스트를 거치게 만드는 작업 흐름이다.

최종 목표는 특정 프로젝트 하나에만 맞는 프롬프트를 만드는 것이 아니다.

Spring Boot, React, Next.js, FastAPI처럼 다양한 프로젝트에서 조합해서 사용할 수 있는 범용 AI 개발 하네스를 만드는 것이 목표다.

모든 프로젝트에 같은 규칙을 강제하는 방식이 아니라, 프로젝트에 필요한 하네스를 골라서 AI 개발팀처럼 구성하는 방식이다.

## 자료를 찾아보며 확인한 것

이 글을 쓰기 전에 하네스와 AI 코딩 agent 관련 자료를 몇 가지 찾아봤다.

찾아보면서 확인한 것은, 내가 느낀 문제가 완전히 혼자만의 문제는 아니라는 점이다.

Anthropic은 장기 실행 애플리케이션 개발에서 하네스 설계가 중요하다고 설명하면서, 작업을 다루기 쉬운 단위로 나누고 구조화된 산출물로 세션 사이의 맥락을 넘기는 방식을 이야기한다. 최근 글에서는 planner, generator, evaluator처럼 역할을 나눈 구조도 소개한다.

OpenAI의 Codex 관련 글과 문서도 비슷한 힌트를 준다. Codex는 `AGENTS.md` 같은 프로젝트 지시문을 읽고, 사용자의 설정과 환경 정보를 합쳐 작업 맥락을 구성한다. 즉, AI가 코드를 잘 만들게 하려면 모델에게 던지는 한 문장만 중요한 것이 아니라, 프로젝트가 어떤 규칙과 맥락을 제공하는지도 중요하다.

Harness Developer Hub의 Harness Skills 문서도 참고할 만했다. 여기서는 상위 지시문이 전체 행동 기준을 잡고, 개별 skill이 생성, 디버깅, 거버넌스, 리포팅 같은 특정 작업을 맡는 구조를 설명한다. 내가 생각하는 PM 하네스와 프레임워크/전문가 하네스의 분리도 이 방향과 비슷하다.

또 `revfactory/harness` 같은 공개 프로젝트도 이미 "domain description -> agent team + skills"라는 방향을 실험하고 있었다. 다만 이런 프로젝트의 성능 수치는 작성자 측 실험인 경우가 많고, 제3자 재현이 충분하지 않을 수 있다. 그래서 이 글에서는 "하네스를 쓰면 코드 품질이 올라간다"고 단정하지 않으려 한다.

정리하면, 내가 세운 가설은 인터넷에서 찾은 자료들과 방향이 맞는다. 하지만 아직 내 프로젝트에서 검증된 것은 아니다.

그래서 이 글의 표현은 계속 가설로 둔다.

> 하네스가 코드 품질을 올린다고 믿는 것이 아니라, 실제 프로젝트에서 그 가설을 검증해보려는 것이다.

## 왜 하네스가 필요하다고 느꼈나

AI 코딩 도구를 쓰면서 가장 먼저 느끼는 장점은 속도다.

하지만 속도가 빠르다는 것은 잘못된 코드도 빠르게 쌓일 수 있다는 뜻이기도 하다.

예를 들어 이런 일이 생길 수 있다.

- 처음 요구사항과 다른 방향으로 기능이 커진다.
- 기존 아키텍처 규칙을 무시한다.
- 테스트 없이 구현부터 한다.
- 서버에서 검증해야 할 값을 프론트엔드 계산값으로 믿는다.
- Controller, Service, Repository 책임이 섞인다.
- 화면은 만들어졌지만 loading, error, empty state가 빠진다.
- 코드가 돌아가는 것처럼 보이지만 실제 브라우저에서 깨진다.
- 세션이 길어질수록 처음 내린 결정이 흐려진다.

사람도 이런 실수를 한다.

다만 AI는 작업 속도가 빠르기 때문에, 문제가 보였을 때 이미 변경 범위가 커져 있을 수 있다.

그래서 AI와 협업할 때는 "더 똑똑한 프롬프트"만으로는 부족하다고 느꼈다.

필요한 것은 작업을 제한하는 구조다.

좋은 AI 협업은 "무엇을 시킬까"보다 "어떤 경계 안에서 일하게 할까"가 더 중요해진다.

## 하네스라는 단어로 부르고 싶은 것

내가 생각하는 하네스는 AI 개발을 위한 작업 통제 구조다.

포함하려는 요소는 대략 이렇다.

- 프로젝트 목표를 정리하는 PM 역할
- 프레임워크별 설계와 구현 기준
- 작업을 작게 나누는 task packet
- worker가 건드릴 수 있는 파일 범위
- 구현 후 리드 리뷰
- 보안, 테스트, 접근성, 클린 코드 게이트
- 최종 PM 리뷰
- 다음 세션을 위한 결정 기록

핵심은 AI가 전체 프로젝트를 매번 새로 판단하지 않게 만드는 것이다.

AI가 할 일을 작게 자르고, 그 작은 일에 필요한 맥락만 주고, 끝난 뒤에는 정해진 기준으로 검토한다.

이 구조가 잘 작동한다면 AI는 더 많은 자유를 얻는 것이 아니라, 더 명확한 책임 안에서 일하게 된다.

## 하나의 큰 프롬프트로 만들지 않으려는 이유

처음에는 모든 규칙을 하나의 큰 문서에 넣으면 될 것처럼 보였다.

예를 들어 이런 식이다.

```text
이 프로젝트는 Spring Boot와 React를 사용한다.
백엔드는 DDD 모듈러 모놀리스로 만든다.
프론트엔드는 feature-based 구조를 따른다.
보안 규칙은 이렇고, 테스트 규칙은 이렇고, UI 규칙은 이렇다.
```

이런 문서가 없는 것보다는 훨씬 낫다.

하지만 모든 규칙을 모든 세션에 넣으면 다른 문제가 생긴다.

프론트엔드 컴포넌트 하나를 수정하는 worker에게 DBA 규칙까지 모두 줄 필요는 없다. 반대로 DB migration을 검토하는 세션에 React 접근성 규칙이 많이 들어갈 필요도 없다.

맥락이 많아질수록 중요한 정보가 선명해지는 것이 아니라 흐려질 수 있다.

그래서 지금 생각하는 방향은 작은 task packet이다.

worker에게 전체 대화를 넘기지 않는다.

worker에게는 필요한 것만 준다.

```text
Role: Backend Worker
Framework: Spring Boot
Architecture: DDD Modular Monolith
Domain: Ecommerce
Security Level: Strict

Goal:
- 주문 생성 유스케이스 구현

Allowed Files:
- src/main/java/.../order/application/CreateOrderService.java
- src/main/java/.../order/domain/Order.java

Rules:
- Controller에서 repository를 직접 호출하지 않는다.
- 가격은 서버에서 다시 계산한다.
- 권한 체크를 생략하지 않는다.

Acceptance:
- 성공 케이스 테스트
- 권한 실패 테스트
- 입력값 검증 실패 테스트
```

worker는 프로젝트를 다시 설계하는 존재가 아니다.

리드가 잘라준 작은 일을 끝내는 존재다.

이렇게 해야 AI가 불필요하게 큰 판단을 하거나, 원래 의도와 다른 구조로 확장하는 일을 줄일 수 있다고 보고 있다.

## PM 하네스가 중심이 된다

전체 구조의 시작점은 PM 하네스다.

PM 하네스는 전역으로 설치하는 유일한 하네스가 되는 것을 목표로 한다.

사용자는 새 프로젝트에서 `$setup`을 실행한다. 그러면 PM 하네스가 프로젝트 목표를 묻고, 필요한 기술 스택을 추천하고, 어떤 하네스 팀이 필요한지 결정한다.

PM 하네스가 맡을 일은 구현 그 자체가 아니다.

오케스트레이션이다.

- 제품 목표 확인
- 사용자 수준 선택
- 프레임워크와 아키텍처 추천
- 필요한 하네스 팀 구성
- GitHub, Notion, Figma, Browser QA 같은 MCP 확인
- `AGENTS.md`, `harness.yaml`, `.harness/decisions.md` 생성
- 개발 전 clear planning gate 실행
- 최종 cross-functional review

여기서 중요한 점은 PM 하네스가 모든 지식을 다 들고 있으면 안 된다는 것이다.

PM은 프로젝트에 필요한 팀을 구성하고, 각 리드에게 일을 나누고, 마지막에 전체 품질을 확인하는 역할에 가깝다.

Spring Boot의 세부 규칙은 Spring Boot 하네스가 더 잘 알아야 한다.

React 화면 구조와 접근성은 React 하네스가 더 잘 알아야 한다.

DB migration 위험은 DBA 하네스가 더 잘 봐야 한다.

## 프로젝트마다 AI 개발팀을 조합한다

내가 만들고 싶은 최종 형태는 모든 프로젝트에 똑같은 프롬프트를 복사하는 방식이 아니다.

프로젝트마다 필요한 하네스를 조합하는 방식이다.

예를 들어 Spring Boot와 React로 만드는 일반적인 웹 서비스라면 이런 팀이 필요할 수 있다.

```text
PM Harness
+ Spring Boot Harness
+ React Harness
+ PostgreSQL DBA Harness
+ QA Harness
= Spring Boot + React 프로젝트용 AI 개발팀
```

UI 중심의 SaaS라면 조합이 달라질 수 있다.

```text
PM Harness
+ Next.js Harness
+ Designer Harness
+ Web Security Harness
+ QA Harness
= UI 중심 SaaS 프로젝트용 AI 개발팀
```

내부 관리자 도구라면 또 다를 수 있다.

```text
PM Harness
+ Spring Boot Harness
+ React Harness
+ Web Security Harness
= Internal Admin 프로젝트용 AI 개발팀
```

핵심은 모든 프로젝트를 하나의 방식으로 강제하지 않는 것이다.

공통 흐름은 유지하되, 필요한 역할과 규칙은 프로젝트에 맞게 선택한다.

그래야 이 하네스가 특정 프로젝트 전용 장난감이 아니라, 여러 프로젝트에 적용 가능한 구조가 될 수 있다.

## 프레임워크 하네스와 전문가 하네스

현재 생각하는 하네스는 크게 두 종류다.

하나는 프레임워크 하네스다.

- `spring-boot-harness`
- `react-harness`
- `nextjs-harness`
- `fastapi-harness`
- `express-harness`

프레임워크 하네스는 각 기술 스택의 팀장 역할을 한다.

예를 들어 Spring Boot 하네스는 Controller, Application, Domain, Infrastructure 경계를 본다. transaction boundary, validation 위치, exception handling, repository 사용 규칙, Spring Security 적용 기준도 본다.

React 하네스는 route, page, component, feature 구조를 본다. API client 경계, form validation, loading/error/empty state, accessibility, XSS 위험, browser QA 기준도 본다.

다른 하나는 전문가 하네스다.

- `postgres-dba-harness`
- `web-security-harness`
- `designer-harness`
- `qa-harness`
- `cloud-harness`
- `ai-harness`
- `devops-harness`

전문가 하네스는 모든 프로젝트에 항상 들어오지 않는다.

DB 정합성이 중요한 프로젝트라면 DBA 하네스가 필요하다. 화면 품질이 중요한 프로젝트라면 designer 하네스가 필요하다. 출시 전 검증이 중요하다면 QA 하네스가 필요하다.

PM 하네스는 프로젝트 성격을 보고 필요한 전문가를 추천해야 한다.

단순 CRUD 프로젝트에 모든 전문가 하네스를 다 불러오면 오히려 무겁다.

필요할 때 필요한 역할만 부르는 것이 목표다.

## `$setup`에서 하고 싶은 일

하네스의 시작점은 `$setup`이다.

상상하는 흐름은 이렇다.

```text
사용자: $setup
-> 안내 수준 선택
-> 제품 아이디어 정리
-> 프레임워크/아키텍처 추천
-> 도메인과 위험도 확인
-> 필요한 MCP 확인
-> 필요한 하네스 repo clone
-> AGENTS.md, harness.yaml, decisions.md 작성
-> 개발 시작 전 clear planning gate
```

여기서 `AGENTS.md`는 프로젝트의 작업 규칙이다.

`harness.yaml`은 기계가 읽을 설정이다.

`.harness/decisions.md`는 사람이 읽을 결정 기록이다.

예를 들어 설정은 이런 식으로 남길 수 있다.

```yaml
backend:
  framework: spring-boot
  architecture: ddd-modular-monolith

frontend:
  framework: react
  architecture: feature-based

quality:
  clean_code: strict
  security: strict
  testing: required

setup:
  guidance_level: beginner
  language: ko
```

이렇게 남겨두면 다음 세션에서 다시 처음부터 설명하지 않아도 된다.

AI는 현재 프로젝트의 선택을 파일에서 다시 읽을 수 있다.

## 왜 한국어 UX까지 고민하나

이 하네스는 나 혼자 쓰는 실험으로 시작하지만, 결국 사람이 선택지를 이해해야 한다.

아키텍처, 보안, DB, 배포, QA 같은 선택지는 초보자에게 추상적으로 느껴질 수 있다.

그래서 `$setup`은 사용자 숙련도에 따라 다르게 설명해야 한다고 생각한다.

- 초보자: 쉬운 한국어 설명, 안전한 기본값, 질문 적게
- 중급자: 중요한 트레이드오프 설명, 핵심 결정만 질문
- 전문가: 짧은 설명, 고급 옵션, custom 허용

예를 들어 백엔드 아키텍처를 고를 때 초보자에게 `ddd-modular-monolith`, `hexagonal`, `cqrs` 같은 값만 보여주면 선택하기 어렵다.

대신 이렇게 설명하는 편이 낫다.

```text
DDD 모듈러 모놀리스
- 비즈니스 규칙이 있는 서비스를 한 애플리케이션 안에서 모듈별로 깔끔하게 나누는 방식입니다.
- 마이크로서비스보다 운영이 단순하고, 나중에 모듈 단위로 분리하기 좋습니다.
```

기계가 읽는 값은 영어로 유지한다.

```yaml
architecture: ddd-modular-monolith
```

하지만 사람이 읽는 설명은 한국어로 제공한다.

하네스가 좋은 선택지를 갖고 있어도, 사용자가 왜 그 선택을 해야 하는지 이해하지 못하면 좋은 도구가 되기 어렵다.

## 지금 MVP는 어디까지인가

처음부터 모든 하네스를 만들 생각은 없다.

지금 생각하는 MVP 순서는 이렇다.

1. `pm-harness`의 기본 skeleton을 만든다.
2. `spring-boot-harness`를 만든다.
3. `react-harness`를 만든다.
4. 작은 Spring Boot + React 샘플 프로젝트를 만든다.
5. PM -> Backend Lead -> Frontend Lead -> Worker -> Lead Review -> PM Review 흐름을 검증한다.
6. 그 다음 `postgres-dba-harness`, `qa-harness`, `designer-harness`, `web-security-harness`를 붙인다.

이미 초기 repository는 만들어둔 상태다.

```text
AI-harness-lab-Lee

pm-harness
spring-boot-harness
react-harness
example-spring-react-app
postgres-dba-harness
web-security-harness
designer-harness
qa-harness
```

다만 아직 "동작하는 도구"라고 말할 단계는 아니다.

지금은 기획과 설계 단계다.

Codex와 계속 대화하면서 어떤 구조가 좋을지 좁혀가고 있고, 이제 첫 skeleton을 만들기 전이다.

## 이 실험에서 확인하고 싶은 것

이 하네스가 정말 의미 있는지는 프롬프트가 멋진지로 판단할 수 없다.

작은 프로젝트 하나를 실제로 끝까지 만들어봐야 한다.

검증하고 싶은 질문은 이런 것들이다.

- 역할을 나누는 방식이 실제 코드 품질을 올리는가?
- 범용 하네스 구조가 특정 프로젝트 전용 규칙보다 재사용성이 높은가?
- PM 하네스만 전역 설치하고 나머지는 프로젝트별로 clone하는 모델이 편한가?
- task packet만으로 worker가 충분히 일할 수 있는가?
- context 사용량을 줄이면서도 구현 품질을 유지할 수 있는가?
- Spring Boot와 React 리드가 worker 결과를 제대로 리뷰할 수 있는가?
- 개발 전 clear planning gate가 AI 작업 품질을 높이는가?
- QA나 DBA 같은 전문가 하네스가 실제로 놓칠 수 있는 문제를 잡아주는가?

만약 이 질문들에 긍정적인 답을 얻는다면, 하네스는 단순한 프롬프트 모음이 아니라 AI 개발 워크플로 자체를 개선하는 도구가 될 수 있다.

반대로 별 효과가 없다면 그 사실도 의미가 있다.

최소한 "AI 코딩 품질을 올리려면 어떤 통제가 필요하고, 어떤 통제는 과한가"를 배울 수 있기 때문이다.

## 마무리

지금 만들고 있는 AI 개발 하네스는 아직 완성된 도구가 아니다.

정확히 말하면 가설을 검증하기 위한 실험에 가깝다.

AI에게 큰 작업을 통째로 맡기는 것보다, PM, 리드, 전문가, worker로 역할을 나누고, 작은 task packet과 리뷰 게이트를 두면 코드 품질과 일관성이 올라갈 수 있을까?

이 질문에 답하기 위해 하네스를 만들고 있다.

최종 목표는 특정 프로젝트 하나에만 맞는 프롬프트가 아니다.

여러 프로젝트에 조합해서 적용할 수 있는 범용 AI 개발 하네스다.

Spring Boot 프로젝트에는 Spring Boot 리드를 붙이고, React 프로젝트에는 React 리드를 붙이고, DB 위험이 큰 프로젝트에는 DBA를 붙이고, 출시 전에는 QA를 붙이는 식이다.

사람 개발팀이 역할을 나누고 리뷰를 거치듯, AI 개발도 그런 구조를 가질 수 있는지 확인해보고 싶다.

이 하네스가 성공했는지는 말로 판단할 수 없다.

작은 프로젝트 하나를 실제로 끝까지 만들고, 기획부터 구현, 리뷰, 검증까지 흐름이 무너지지 않는지 확인해야 한다.

지금은 그 첫 번째 실험을 준비하는 단계다.
