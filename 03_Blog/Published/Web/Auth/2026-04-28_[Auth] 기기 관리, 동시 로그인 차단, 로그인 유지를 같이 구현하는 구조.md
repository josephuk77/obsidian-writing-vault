# [Auth] 기기 관리, 동시 로그인 차단, 로그인 유지를 같이 구현하는 구조

로그인 기능을 구현할 때 처음에는 단순히 JWT만 발급하면 끝이라고 생각할 수 있다.
하지만 실제 서비스에서는 단순 로그인뿐만 아니라 로그인 유지, 여러 기기 관리, 동시 로그인 차단, 특정 기기 로그아웃 같은 기능도 함께 고려해야 한다.

이런 기능을 구현하려면 Access Token 하나만으로는 부족하다.
핵심은 Access Token, Refresh Token, Device ID를 분리해서 관리하는 것이다.
```text
Access Token은 API 요청에 사용한다.
Refresh Token은 로그인 유지와 Access Token 재발급에 사용한다.
Device ID는 사용자의 기기를 구분하는 데 사용한다.
```
## Access Token과 Refresh Token의 역할

Access Token은 일반 API 요청에 사용하는 토큰이다.
```http
GET /api/posts
Authorization: Bearer accessToken
```
Access Token은 노출되었을 때 위험하기 때문에 수명을 짧게 가져가는 것이 좋다.
보통 프론트에서는 localStorage나 cookie에 오래 저장하지 않고, 메모리에만 저장한다.

반면 Refresh Token은 Access Token을 다시 발급받기 위한 토큰이다.
Access Token보다 수명이 길고, 사용자가 로그인 상태를 유지할 수 있게 해준다.

정리하면 다음과 같다.
```text
Access Token
- 일반 API 요청에 사용
- 수명이 짧음
- 프론트 메모리에 저장

Refresh Token
- Access Token 재발급에 사용
- 수명이 김
- HttpOnly Cookie에 저장
- 서버 DB 또는 Redis에도 저장
```
여기서 중요한 점은 로그인 유지의 핵심은 Access Token이 아니라 Refresh Token이라는 것이다.

## 로그인 유지가 되는 이유

사용자가 로그인하면 서버는 Access Token과 Refresh Token을 발급한다.

Access Token은 API 요청에 사용되다가 일정 시간이 지나면 만료된다.
이때 프론트는 /auth/reissue 같은 재발급 API를 호출한다.
```text
API 요청
→ Access Token 만료
→ 401 응답
→ Refresh Token으로 Access Token 재발급
→ 기존 요청 재시도
```
이 과정이 자동으로 처리되면 사용자는 Access Token이 만료되었다는 사실을 거의 느끼지 못한다.
결국 사용자는 계속 로그인되어 있는 것처럼 느끼지만, 실제로는 Access Token이 계속 새로 발급되고 있는 것이다.

또한 새로고침을 하면 메모리에 저장된 Access Token은 사라진다.
하지만 Refresh Token이 HttpOnly Cookie에 남아 있다면, 앱이 다시 실행될 때 재발급 요청을 보내서 로그인 상태를 복원할 수 있다.
```text
새로고침
→ Access Token 사라짐
→ 앱 시작 시 /auth/reissue 호출
→ Refresh Token 쿠키 전송
→ 새 Access Token 발급
→ 로그인 상태 복원
```
## Refresh Token은 왜 쿠키에 저장할까?

Refresh Token은 오래 살아 있는 중요한 토큰이다.
그래서 JavaScript로 쉽게 접근할 수 있는 localStorage에 저장하는 것은 위험할 수 있다.

대신 보통은 HttpOnly, Secure 옵션을 적용한 쿠키에 저장한다.
```http
Set-Cookie: refreshToken=...; HttpOnly; Secure; SameSite=Lax; Path=/auth/reissue
```
HttpOnly 옵션을 사용하면 JavaScript에서 쿠키 값을 직접 읽을 수 없다.
즉, XSS 공격이 발생하더라도 Refresh Token이 바로 탈취될 가능성을 줄일 수 있다.

또한 Path=/auth/reissue처럼 경로를 제한하면 Refresh Token 쿠키가 모든 API 요청에 붙지 않고, 재발급 요청에만 전송되도록 만들 수 있다.
```http
GET /api/posts
→ Refresh Token 쿠키 전송 안 됨

POST /auth/reissue
→ Refresh Token 쿠키 전송
```
Refresh Token은 중요한 값이기 때문에 최대한 필요한 순간에만 사용되도록 제한하는 것이 좋다.

## Device ID로 기기를 구분하기

기기 관리 기능을 만들려면 사용자가 어떤 기기에서 로그인했는지 구분할 수 있어야 한다.
이를 위해 Device ID를 사용한다.

Device ID는 휴대폰이나 노트북의 실제 고유번호가 아니다.
서버가 랜덤으로 생성한 값이다.
```text
처음 로그인
→ 서버가 deviceId 생성
→ deviceId를 쿠키에 저장
→ 이후 요청에서 같은 deviceId 확인
```
이 Device ID를 기준으로 user_devices 테이블에 기기 정보를 저장할 수 있다.

예를 들어 다음과 같은 정보를 저장할 수 있다.
```text
userId
deviceId
deviceName
browser
os
lastIp
lastSeenAt
revokedAt
```
이 정보를 활용하면 사용자에게 현재 로그인된 기기 목록을 보여줄 수 있다.

## 동시 로그인 차단 원리

동시 로그인을 막고 싶다면 새 기기에서 로그인할 때 기존 Refresh Token을 폐기하면 된다.

예를 들어 휴대폰에서 이미 로그인되어 있다고 하자.
```text
휴대폰 로그인
Refresh Token: R_phone
```
이후 컴퓨터에서 새로 로그인하면 서버는 기존 휴대폰의 Refresh Token을 폐기하고, 컴퓨터용 Refresh Token을 새로 저장한다.
```text
컴퓨터 로그인
→ 기존 휴대폰 Refresh Token 폐기
→ 새 컴퓨터 Refresh Token 저장
```
그러면 휴대폰은 더 이상 Access Token을 재발급받을 수 없다.

다만 휴대폰이 즉시 로그아웃되지 않을 수도 있다.
왜냐하면 휴대폰이 가지고 있던 Access Token이 아직 만료되지 않았을 수 있기 때문이다.
```text
기존 Access Token이 아직 유효함
→ 잠깐 동안 API 요청 가능

Access Token 만료
→ Refresh Token으로 재발급 시도
→ Refresh Token이 폐기되어 실패
→ 로그아웃 처리
```
그래서 동시 로그인 차단을 구현해도 기존 기기가 바로 튕기지 않고, 잠시 뒤 로그아웃되는 것처럼 보일 수 있다.

즉시 차단이 필요하다면 Access Token 블랙리스트를 추가로 사용해야 한다.
하지만 이 경우 매 요청마다 블랙리스트를 확인해야 하므로 구조가 조금 더 복잡해진다.

여러 기기 로그인을 허용하는 경우

반대로 여러 기기 로그인을 허용하고 싶다면 기존 Refresh Token을 폐기하지 않으면 된다.
```text
휴대폰 Refresh Token 유지
컴퓨터 Refresh Token 추가 저장
태블릿 Refresh Token 추가 저장
```
이렇게 하면 사용자는 휴대폰, 컴퓨터, 태블릿에서 모두 로그인 상태를 유지할 수 있다.

이 방식은 사용성이 좋다.
하지만 기기 관리 화면, 비정상 로그인 감지, 전체 로그아웃 기능 같은 보안 기능이 더 중요해진다.

서비스 정책에 따라 다음 중 하나를 선택할 수 있다.
```text
1. 한 기기만 로그인 허용
- 새 로그인 시 기존 Refresh Token 전부 폐기
- 계정 공유 차단에 좋음
- 사용자는 불편할 수 있음

2. 여러 기기 로그인 허용
- 기기별 Refresh Token 유지
- 사용성이 좋음
- 기기 관리 기능이 중요함

3. 최대 기기 수 제한
- 예: 최대 3대까지 로그인 허용
- 초과 시 오래된 기기 로그아웃
- 보안과 사용성의 중간 방식
```
## Refresh Token은 서버에도 저장해야 한다

JWT를 사용한다고 해서 서버에 아무것도 저장하지 않으면 기기 관리나 강제 로그아웃을 구현하기 어렵다.

특히 Refresh Token은 서버 DB나 Redis에 저장해두는 것이 좋다.
그래야 특정 기기의 Refresh Token을 폐기하거나, 전체 로그아웃을 처리할 수 있다.

다만 서버에는 Refresh Token 원문을 그대로 저장하지 않는 것이 좋다.
```text
브라우저 쿠키
→ Refresh Token 원문 저장

서버 DB
→ Refresh Token 해시값 저장

```
DB가 유출되더라도 Refresh Token 원문이 바로 노출되지 않도록 하기 위해서다.

Refresh Token 테이블에는 보통 이런 값들이 들어간다.
```text
userId
deviceId
tokenHash
expiresAt
absoluteExpiresAt
createdAt
lastUsedAt
revokedAt
```
revokedAt이 null이면 아직 유효한 토큰이고, 값이 들어 있으면 폐기된 토큰으로 볼 수 있다.

## Refresh Token Rotation

Refresh Token Rotation은 Refresh Token을 사용할 때마다 새 Refresh Token으로 교체하는 방식이다.
```text
기존 Refresh Token 사용
→ 기존 Refresh Token 폐기
→ 새 Access Token 발급
→ 새 Refresh Token 발급
→ 새 Refresh Token 쿠키 저장
```
이 방식의 장점은 탈취된 Refresh Token이 재사용되는 상황을 감지할 수 있다는 것이다.

예를 들어 이미 한 번 사용되어 폐기된 Refresh Token이 다시 들어오면, 서버는 이를 비정상적인 재사용으로 판단할 수 있다.
```text
폐기된 Refresh Token 재사용 감지
→ 토큰 탈취 가능성 있음
→ 해당 사용자의 모든 Refresh Token 폐기
→ 재로그인 요구

```
다만 Rotation을 사용할 때는 프론트에서도 주의해야 한다.
여러 API 요청이 동시에 401을 받으면 각각 재발급 요청을 보내는 문제가 생길 수 있다.

첫 번째 재발급 요청이 Refresh Token을 교체한 뒤, 두 번째 요청이 예전 Refresh Token을 다시 사용하면 서버는 이를 재사용 공격으로 오해할 수 있다.

그래서 프론트에서는 재발급 요청이 동시에 여러 번 나가지 않도록 잠금 처리를 해주는 것이 좋다.

## 기기 관리 기능

기기 관리 화면을 만들려면 user_devices와 refresh_tokens 정보를 활용하면 된다.

사용자에게는 다음과 같은 정보를 보여줄 수 있다.
```json
[
  {
    "deviceName": "iPhone Safari",
    "browser": "Safari",
    "os": "iOS",
    "lastIp": "123.123.123.123",
    "lastSeenAt": "2026-04-28T20:10:00"
  },
  {
    "deviceName": "Chrome Windows",
    "browser": "Chrome",
    "os": "Windows",
    "lastIp": "111.111.111.111",
    "lastSeenAt": "2026-04-28T21:30:00"
  }
]
```
사용자가 특정 기기를 로그아웃하면 해당 deviceId에 연결된 Refresh Token을 폐기하면 된다.
```text
특정 기기 로그아웃
→ 해당 deviceId의 Refresh Token 폐기
→ 해당 기기는 Access Token 만료 후 재발급 실패
→ 로그아웃 처리
```
전체 로그아웃은 사용자의 모든 Refresh Token을 폐기하는 방식으로 구현할 수 있다.
```text
전체 로그아웃
→ 해당 userId의 모든 Refresh Token 폐기
→ 모든 기기에서 재발급 실패
→ 모든 기기 로그아웃
```
이 기능은 비밀번호 변경, 계정 탈취 의심, 사용자가 직접 모든 기기에서 로그아웃하고 싶을 때 유용하다.

최종 구조 정리

기기 관리, 로그인 유지, 동시 로그인 차단을 함께 구현하려면 다음 구조가 가장 이해하기 쉽다.
```text
Access Token
- JWT
- 짧은 수명
- 프론트 메모리에 저장
- API 요청 시 Authorization 헤더로 전송

Refresh Token
- 긴 수명
- HttpOnly Secure Cookie 저장
- 서버에는 해시값 저장
- Access Token 재발급에 사용
- 기기별로 관리

Device ID
- 서버가 랜덤 생성
- 쿠키에 저장
- user_devices 테이블과 연결
- 기기 목록 조회와 특정 기기 로그아웃에 사용

동시 로그인 차단
- 새 로그인 시 기존 Refresh Token 폐기

여러 기기 로그인 허용
- 기기별 Refresh Token을 따로 저장

즉시 로그아웃
- Access Token 만료 시간을 짧게 설정
- 필요하면 Access Token 블랙리스트 사용
```
## 결론

로그인 유지의 핵심은 Refresh Token이다.
Access Token은 짧게 사용하고, 만료되면 Refresh Token으로 다시 발급받는다.

기기 관리의 핵심은 Device ID다.
서버가 발급한 Device ID를 기준으로 각 기기의 로그인 상태를 관리할 수 있다.

동시 로그인 차단의 핵심은 기존 Refresh Token 폐기다.
새 기기에서 로그인할 때 기존 기기의 Refresh Token을 폐기하면, 기존 기기는 Access Token이 만료된 뒤 더 이상 재발급을 받지 못하고 로그아웃된다.

정리하면 다음과 같다.
```text
Access Token은 짧게
Refresh Token은 길게
Refresh Token은 서버에서 관리
기기는 Device ID로 구분
동시 로그인 차단은 기존 Refresh Token 폐기로 처리
```
결국 실제 서비스의 로그인 구조는 단순히 JWT 하나를 발급하는 것이 아니라,
Access Token, Refresh Token, Device ID를 각각의 역할에 맞게 나누어 관리하는 구조라고 볼 수 있다.
