# [Database] Redis vs DB, 저장소의 목적부터 다르다

Redis를 공부하다 보면 DB와 비교하는 이야기를 자주 보게 된다.

처음에는 Redis도 데이터를 저장하니까 DB의 한 종류처럼 생각하기 쉽다. 실제로 Redis는 값을 저장하고 조회할 수 있고, 설정에 따라 디스크에 데이터를 남길 수도 있다.

그런데 프로젝트에서 Redis를 쓰는 위치를 보면 일반적인 RDBMS와는 역할이 조금 다르다.

DB에는 주문, 결제, 회원, 게시글 같은 핵심 데이터를 저장한다. 반면 Redis에는 캐시, 세션, 인증 토큰, 조회수, 랭킹, 중복 요청 방지 키 같은 데이터를 저장하는 경우가 많다.

이번 글에서는 Redis와 DB를 "빠르다 / 느리다"로 비교하기보다, 저장소의 목적과 보장 범위가 어떻게 다른지 정리해보려고 한다.

## Redis와 DB를 왜 비교하게 될까

백엔드 프로젝트를 만들다 보면 처음에는 DB만 사용한다.

회원 정보를 저장하고, 게시글을 저장하고, 주문 데이터를 저장한다.

그러다가 성능이나 동시성 문제를 만나면 Redis가 등장한다.

예를 들면 이런 상황이다.

- 로그인 세션을 Redis에 저장한다
- 게시글 조회수를 Redis에서 증가시킨다
- 자주 조회되는 상품 정보를 Redis에 캐싱한다
- 같은 쿠폰을 여러 번 발급하지 않기 위해 Redis에 키를 저장한다
- 여러 서버에서 동시에 같은 작업을 하지 않도록 Redis Lock을 사용한다

이런 경험이 쌓이면 자연스럽게 이런 질문이 생긴다.

```text
Redis도 데이터를 저장하는데 DB처럼 써도 되는 것 아닐까?
```

결론부터 말하면, Redis를 DB처럼 사용할 수 있는 경우도 있지만 일반적인 RDBMS와 같은 역할을 기대하면 위험할 수 있다.

Redis와 DB는 데이터를 저장한다는 공통점은 있지만, 기본적으로 잘하는 일이 다르다.

## DB가 강한 영역

DB는 데이터 정합성이 중요한 곳에 강하다.

예를 들어 주문 기능을 생각해보자.

사용자가 상품을 주문하면 보통 하나의 작업만 일어나지 않는다.

- 주문 데이터 저장
- 상품 재고 감소
- 결제 내역 저장
- 포인트 사용 내역 저장

이 작업들은 따로 성공하면 안 된다.

주문은 생성됐는데 재고가 줄지 않거나, 결제 내역은 저장됐는데 주문이 없는 상태가 되면 데이터가 깨진다.

그래서 DB에서는 이런 작업을 트랜잭션으로 묶는다.

```sql
BEGIN;

UPDATE product
SET stock = stock - 1
WHERE id = 1;

INSERT INTO orders(product_id, quantity)
VALUES (1, 1);

COMMIT;
```

정상적으로 끝나면 `COMMIT`하고, 중간에 문제가 생기면 `ROLLBACK`한다.

이런 구조 덕분에 여러 SQL 작업을 하나의 작업처럼 다룰 수 있다.

DB가 중요한 이유는 단순히 데이터를 저장하기 때문이 아니다.

데이터가 잘못된 상태로 남지 않도록 트랜잭션, 제약 조건, Lock, 로그 같은 장치를 제공하기 때문이다.

정리하면 이렇게 볼 수 있다.

> DB는 원본 데이터의 정합성과 영속성을 지키는 저장소다.

## Redis가 강한 영역

Redis는 메모리 기반 저장소다.

그래서 디스크 기반 DB보다 훨씬 빠르게 값을 읽고 쓸 수 있다.

또한 Redis는 단순한 key-value 저장소를 넘어서 여러 자료구조를 제공한다.

- String
- List
- Set
- Sorted Set
- Hash

이런 자료구조 덕분에 단순 캐시뿐만 아니라 랭킹, 큐, 카운터 같은 기능도 쉽게 만들 수 있다.

예를 들어 게시글 조회수를 증가시키는 작업은 Redis에 잘 맞는다.

```redis
INCR post:1:view_count
```

`INCR` 명령은 값을 1 증가시킨다.

이 명령 하나는 Redis에서 원자적으로 처리된다. 여러 요청이 동시에 들어와도 `INCR` 명령 하나가 중간에 쪼개져 실행되지는 않는다.

인증 토큰처럼 일정 시간이 지나면 사라져야 하는 값도 Redis에 잘 맞는다.

```redis
SET login:token:abc user:1 EX 3600
```

이렇게 저장하면 3600초 뒤에 키가 만료된다.

DB에서도 만료 시간을 컬럼으로 두고 처리할 수는 있지만, Redis는 TTL이 기본 기능으로 제공되기 때문에 임시 데이터 관리에 편하다.

Redis가 잘 맞는 대표적인 상황은 다음과 같다.

- 캐시
- 로그인 세션
- 인증 토큰
- 조회수 카운터
- 실시간 랭킹
- 중복 요청 방지
- 짧은 시간만 필요한 임시 상태

정리하면 이렇게 볼 수 있다.

> Redis는 빠른 접근, TTL, 단순한 원자 연산, 자료구조 활용이 중요한 곳에 강하다.

## Redis를 DB처럼 쓰면 안 되는 걸까

"무조건 안 된다"라고 말할 수는 없다.

Redis도 데이터를 저장할 수 있고, persistence 설정을 통해 데이터를 디스크에 남길 수도 있다.

하지만 일반적인 웹 애플리케이션에서 Redis를 RDBMS처럼 핵심 원본 데이터 저장소로 사용하는 것은 조심해야 한다.

이유는 기대하는 보장 범위가 다르기 때문이다.

DB에서는 보통 이런 것을 기대한다.

- 여러 작업을 하나의 트랜잭션으로 묶는다
- 실패하면 롤백한다
- 제약 조건으로 잘못된 데이터를 막는다
- 커밋된 데이터가 장애 이후에도 남는다
- 동시에 접근해도 정합성이 깨지지 않도록 제어한다

Redis에도 원자적 명령이나 트랜잭션 기능이 있지만, RDBMS의 트랜잭션과 같은 모델은 아니다.

특히 Redis 트랜잭션은 `EXEC` 이후 실행 중 에러가 난 명령을 자동으로 롤백하지 않는다.

그래서 Redis를 사용할 때는 이렇게 역할을 나누는 경우가 많다.

```text
DB: 원본 데이터 저장
Redis: 빠른 조회, 임시 상태, 캐시, 카운터, 중복 방지
```

예를 들어 게시글 조회수는 Redis에서 빠르게 증가시키고, 일정 주기마다 DB에 반영할 수 있다.

하지만 주문, 결제, 재고처럼 틀리면 안 되는 데이터는 DB를 최종 원본으로 두는 것이 일반적이다.

## 예시로 보는 역할 분리

상품 상세 페이지를 생각해보자.

상품 정보는 DB에 저장되어 있다.

```text
product
- id
- name
- price
- stock
```

사용자가 상품 상세 페이지에 들어올 때마다 DB를 조회하면 트래픽이 많을 때 부담이 될 수 있다.

그래서 Redis에 캐시를 둘 수 있다.

```text
product:1:detail -> 상품 상세 JSON
```

조회 흐름은 이렇게 된다.

```text
1. Redis에서 product:1:detail 조회
2. 있으면 Redis 값을 반환
3. 없으면 DB에서 조회
4. 조회한 값을 Redis에 저장
5. 사용자에게 반환
```

이 경우 Redis는 빠른 조회를 위한 보조 저장소다.

원본 데이터는 DB에 있다.

만약 Redis 값이 사라져도 DB에서 다시 만들 수 있다.

반대로 주문 데이터는 다르다.

주문 데이터가 Redis에서 사라졌을 때 "DB에서 다시 만들면 되지"라고 말할 수 없다. 주문은 원본 데이터 자체이기 때문이다.

그래서 주문, 결제, 재고 같은 데이터는 DB 트랜잭션을 중심으로 설계하는 것이 안전하다.

## Redis와 DB의 차이를 한 번에 정리하면

| 기준 | DB | Redis |
| --- | --- | --- |
| 주된 목적 | 원본 데이터 저장 | 빠른 조회와 임시 상태 관리 |
| 강점 | 정합성, 영속성, 트랜잭션 | 속도, TTL, 자료구조, 원자 명령 |
| 대표 데이터 | 주문, 결제, 회원, 재고 | 캐시, 세션, 토큰, 랭킹, 카운터 |
| 실패 시 관점 | 데이터가 깨지면 안 됨 | 사라져도 복구 가능한 값이 많음 |
| 트랜잭션 | `COMMIT`, `ROLLBACK` 중심 | `MULTI`, `EXEC` 중심 |
| 사용 기준 | 최종 원본 데이터 | 빠르게 접근할 보조 데이터 |

물론 실제 시스템에서는 더 복잡하다.

Redis를 primary storage처럼 사용하는 경우도 있고, DB도 캐시처럼 쓰는 경우가 있다.

하지만 백엔드 입문 단계에서는 먼저 이 기준을 잡는 것이 좋다.

```text
정합성이 가장 중요한 원본 데이터는 DB
빠른 접근과 임시 상태가 중요한 데이터는 Redis
```

## 내가 헷갈렸던 부분

처음에는 Redis가 빠른 DB라고 생각했다.

그래서 Redis를 쓰면 DB보다 무조건 좋은 것처럼 느껴졌다.

하지만 공부해보니 Redis와 DB는 단순히 성능 차이만 있는 것이 아니었다.

DB는 느리지만 안정적인 저장소이고, Redis는 빠르지만 가볍게 쓰는 저장소라고 단순화하면 또 틀릴 수 있다.

더 정확하게는 이렇게 이해하는 것이 좋다.

> DB와 Redis는 서로 대체 관계라기보다, 서로 다른 책임을 맡는 저장소다.

DB는 원본 데이터의 정합성을 책임지고, Redis는 빠른 접근과 임시 상태 관리를 도와준다.

이 차이를 이해해야 Redis를 어디에 써야 하는지 판단할 수 있다.

## 정리

Redis와 DB는 모두 데이터를 저장할 수 있다.

하지만 저장소의 목적은 다르다.

DB는 주문, 결제, 재고처럼 데이터 정합성과 영속성이 중요한 곳에 적합하다. 트랜잭션, 롤백, 제약 조건, Lock 같은 기능을 통해 원본 데이터를 안정적으로 관리한다.

Redis는 캐시, 세션, 토큰, 카운터, 랭킹처럼 빠른 접근과 TTL이 중요한 곳에 적합하다. 개별 명령의 원자성과 다양한 자료구조를 활용해 빠르게 데이터를 처리할 수 있다.

그래서 Redis를 사용할 때는 이 기준을 먼저 생각하면 좋다.

```text
이 데이터가 사라져도 DB에서 다시 만들 수 있는가?
이 데이터가 틀리면 비즈니스적으로 큰 문제가 생기는가?
정합성의 최종 책임을 어디에 둘 것인가?
```

이 질문에 대한 답이 "절대 틀리면 안 되는 원본 데이터"라면 DB를 중심으로 설계하는 것이 좋다.

반대로 빠른 조회, 임시 상태, 중복 방지, 카운터처럼 보조적인 역할이라면 Redis가 좋은 선택이 될 수 있다.
