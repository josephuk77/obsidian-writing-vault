# [Spring AI] 프롬프트, 메시지 구조, 그리고 프롬프트 엔지니어링 정리

이번에는 Spring AI를 사용하면서 가장 헷갈렸던 부분인  
PromptTemplate, Message 구조, 그리고 프롬프트 엔지니어링 개념들을 정리해보려고 한다.

앞에서 ChatModel과 ChatClient로 챗봇을 구현해봤다면  
이번 내용은 그 내부에서 “LLM이 어떻게 동작하는지”를 이해하는 단계라고 보면 된다.

---

# PromptTemplate에서 create, render, createMessage 차이

Spring AI에서 PromptTemplate은 단순 문자열 치환 도구가 아니라  
LLM에 전달할 프롬프트 구조를 만들어주는 역할을 한다.

## render()
```java
PromptTemplate template = new PromptTemplate("안녕 {name}");
String result = template.render(Map.of("name", "승욱"));
```
결과는 단순 문자열이다.
```text
안녕 승욱
```
이건 디버깅 용도로 사용한다고 보면 된다.  
실제 LLM 호출보다는 프롬프트가 어떻게 만들어지는지 확인할 때 사용한다.

---

## create()
```java
Prompt prompt = template.create(Map.of("name", "승욱"));
```
Prompt 객체를 생성한다.  
ChatModel에 바로 넣을 수 있는 형태다.
```java
chatModel.call(prompt);
```
실제 LLM 호출에서는 이걸 가장 많이 사용한다.

---

## createMessage()
```java
Message message = template.createMessage(Map.of("name", "승욱"));
```
Message 하나를 생성한다.  
보통 UserMessage로 만들어진다.

여러 메시지를 직접 조합할 때 사용한다.

---

## 정리

- render = 문자열
- createMessage = 메시지 1개
- create = LLM 호출용 전체 프롬프트

---

# Message 구조 이해하기

Spring AI에서 가장 중요한 구조는 다음 세 가지다.

- SystemMessage
- UserMessage
- AssistantMessage

이 세 가지로 대화가 구성된다.

---

## SystemMessage
```java
new SystemMessage("너는 시니어 백엔드 개발자야. 짧고 핵심만 설명해.");
```
AI의 역할과 스타일을 정의한다.  
프롬프트 엔지니어링의 핵심이라고 볼 수 있다.

---

## UserMessage
```java
new UserMessage("DI가 뭐야?");
```
사용자의 질문이다.  
프론트에서 들어오는 입력이 그대로 들어간다.

---

## AssistantMessage
```java
new AssistantMessage("DI는 의존성 주입이야");
```
AI의 이전 답변이다.  
대화를 이어가기 위해 필요하다.

---

# 메시지 순서가 중요한 이유

LLM은 위에서 아래로 읽는다.  
그리고 뒤쪽에 있는 내용일수록 영향력이 크다.

정석 구조는 다음과 같다.
```text
System
User
Assistant
User
Assistant
현재 User
```
코드로 보면
```java
List<Message> messages = new ArrayList<>();

messages.add(new SystemMessage("너는 친절한 AI야"));
messages.add(new UserMessage("DI 뭐야?"));
messages.add(new AssistantMessage("DI는 의존성 주입이야"));
messages.add(new UserMessage("왜 써?"));
```
중요한 규칙은 세 가지다.

- SystemMessage는 항상 맨 위
- 대화는 시간 순서대로
- 현재 질문은 항상 마지막

---

# String으로 넣는 것과 Message로 넣는 것의 차이

처음에는 그냥 문자열로 넣어도 되는 줄 알았다.
```java
String prompt = "너는 친절한 AI야. DI가 뭐야?";
```
이 방식은 역할 구분이 안 된다.

LLM 입장에서는  
이게 지시인지 질문인지 애매하다.

---

## Message 방식
```java
List<Message> messages = List.of(
    new SystemMessage("너는 친절한 AI야"),
    new UserMessage("DI가 뭐야?")
);
```
이건 역할이 명확하다.

- system = 규칙
- user = 질문

모델이 학습된 구조 그대로라서 더 정확하게 동작한다.

---

# 복수 메시지 추가 방법

여러 메시지는 List에 넣으면 된다.
```java
List<Message> messages = new ArrayList<>();

messages.add(new SystemMessage("너는 강사야"));
messages.add(new UserMessage("DI 뭐야?"));
messages.add(new AssistantMessage("DI는 의존성 주입이야"));
messages.add(new UserMessage("왜 써?"));
```
핵심은 순서다.

---

# 대화 기억은 어떻게 구현되는가

LLM은 상태를 기억하지 않는다.  
그래서 매번 전체 대화를 다시 보내야 한다.
```java
List<Message> history = new ArrayList<>();

history.add(new SystemMessage("너는 친절한 AI야"));

history.add(new UserMessage("DI 뭐야?"));
String answer1 = call(history);
history.add(new AssistantMessage(answer1));

history.add(new UserMessage("왜 써?"));
String answer2 = call(history);
history.add(new AssistantMessage(answer2));
```
이렇게 계속 쌓는다.

---

## 메모리 관리 문제

계속 쌓으면 토큰이 제한에 걸린다.

그래서 보통 다음 방법을 사용한다.

- 최근 N개만 유지
- 이전 대화 요약
- Redis / DB 저장

Spring AI에서는 MessageWindowChatMemory 같은 기능도 제공한다.

---

# 프롬프트 엔지니어링 개념 정리

이 부분이 가장 중요하다.  
단순히 질문을 던지는 게 아니라  
“AI가 생각하는 방식”을 설계하는 것이다.

---

## 제로샷 (Zero-shot)

예시 없이 바로 질문
```text
DI 설명해줘
```
간단하지만 정확도는 상황에 따라 다르다.

---

## 퓨샷 (Few-shot)

예시를 몇 개 제공
```text
입력: 2 + 2
출력: 4

입력: 3 + 5
출력: 8

입력: 10 + 7
출력:
```
패턴을 학습시키는 방식이다.

---

## 역할 부여 (Role Prompting)
```text
너는 시니어 백엔드 개발자야
```
SystemMessage로 설정하는 것이 정석이다.

---

## 스탭백 (Step-back Prompting)

문제를 바로 풀지 않고 한 단계 뒤에서 생각하게 한다.
```text
이 문제의 본질은 무엇인가?
어떤 개념이 필요한가?
```
설계 문제나 개념 이해에 효과적이다.

---

## 생각의 사슬 (Chain of Thought)

단계별로 생각하게 한다.
```text
단계별로 생각해서 설명해줘
```
복잡한 문제 해결에 강하다.

---

## 자기 일관성 (Self-consistency)

여러 번 답을 생성하고 가장 일관된 답을 선택한다.  
정확도는 높지만 비용이 크다.

---

# 실무에서의 조합

실제로는 하나만 쓰지 않는다.
```text
System:
너는 시니어 백엔드 개발자야

User:
단계별로 생각해서 설명하고,
먼저 개념을 정리한 후 답변해줘
```
이렇게 역할 부여 + CoT + Step-back을 같이 사용한다.

---

# 정리

Spring AI에서 중요한 핵심은 다음 세 가지다.

- Message 구조 이해
- 메시지 순서 유지
- 대화 히스토리 관리

이걸 기반으로 프롬프트 엔지니어링까지 적용하면  
단순한 챗봇이 아니라 “의도한 답을 만들어내는 시스템”을 만들 수 있다.
