# FaithLog 서비스 기획서

## 서비스 개요

FaithLog는 교회 캠퍼스 내 경건생활 입력, 예배 및 모임 참여 투표, 점심 투표, 벌금 정산, 납부 상태 관리, 푸시 알림을 통합 제공하는 모바일 앱 서비스이다.

사용자는 앱을 통해 한 주 동안의 큐티, 기도, 말씀읽기 일수를 입력하고, 수요예배와 토요 목자모임 참여 여부에 응답하며, 점심 투표에 참여할 수 있다.

관리자는 앱 내 관리자 화면을 통해 구성원들의 입력 여부, 투표 응답 여부, 벌금 정산 결과, 납부 요청 상태를 확인하고 관리할 수 있다.

경건생활 입력은 매주 일요일에 진행한다. 사용자가 일요일에 한 주 동안의 큐티, 기도, 말씀읽기 수행 일수와 토요 목자모임 지각 시간을 입력하면, 시스템은 기준 일수에서 부족한 일수와 지각 시간을 기준으로 벌금을 자동 계산한다.

매주 월요일에는 시스템이 경건생활 미입력자와 벌금 미납자를 자동으로 확인한다. 경건생활을 입력하지 않은 사용자에게는 입력 요청 알림을 보내고, 벌금이 발생했지만 납부 완료 상태가 아닌 사용자에게는 납부 요청 알림을 보낸다.

사용자가 이미 납부 요청을 보낸 상태라면 사용자에게 미납 알림을 보내지 않고, 관리자에게 납부 확인 요청 알림을 보낸다.

초기 서비스는 우리 캠퍼스 기준으로 운영 가능한 형태로 개발하되, 데이터 구조는 향후 여러 캠퍼스에서도 사용할 수 있도록 캠퍼스와 그룹 단위로 분리하여 설계한다.

---

## 서비스 구조

FaithLog는 모바일 앱, 백엔드 서버, 데이터베이스, 푸시 알림 시스템으로 구성된다.

모바일 앱은 React Native로 개발한다.

백엔드 서버는 Spring Boot 기반 REST API 서버로 개발한다.

데이터베이스는 Supabase PostgreSQL을 사용한다.

푸시 알림은 Firebase Cloud Messaging을 사용한다.

앱은 데이터베이스에 직접 접근하지 않고, 모든 요청은 Spring Boot API 서버를 통해 처리한다.

서비스 구조는 다음과 같다.

```
React Native 앱
→ Spring Boot REST API 서버
→ Supabase PostgreSQL
→ Firebase Cloud Messaging
```

Supabase는 인증이나 클라이언트 직접 연동 용도가 아니라, PostgreSQL 데이터베이스 용도로 사용한다. 인증, 권한, 정산 로직, 알림 로직은 Spring Boot 서버에서 처리한다.

## 기획 배경

교회 캠퍼스에서는 매주 경건생활 체크, 수요예배 참여 여부, 토요 목자모임 참여 여부, 점심 투표, 벌금 정산, 납부 확인이 반복적으로 이루어진다.

경건생활은 큐티, 기도, 말씀읽기를 기준으로 확인하고, 점심 투표는 식당, 메뉴, 밥 양 등 상황에 따라 다른 방식으로 진행된다.

토요 목자모임의 경우 지각 여부와 지각 시간에 따라 벌금이 계산된다.

하지만 이러한 정보가 각각 분리되어 관리되면 운영자는 매번 미입력자, 미응답자, 벌금 대상자, 납부 여부를 직접 확인해야 한다.

이 과정에서 누락이 발생할 수 있고, 벌금 계산이나 납부 상태 확인에 불필요한 시간이 소요된다.

또한 구성원들이 경건생활 입력이나 투표 응답, 벌금 납부를 놓치는 경우가 있어 반복적인 안내가 필요하다.

FaithLog는 이러한 반복적인 운영 업무를 모바일 앱과 푸시 알림으로 통합하여, 구성원은 쉽게 입력하고 응답하며, 관리자는 입력 현황과 정산 상태를 효율적으로 관리할 수 있도록 돕기 위해 기획되었다.

## 해결하려는 문제

경건생활 입력 여부를 한눈에 확인하기 어렵다.

수요예배와 토요 목자모임 참여 여부를 매주 반복해서 확인해야 한다.

점심 투표는 식당, 메뉴, 밥 양 등 매번 형태가 달라 관리가 번거롭다.

큐티, 기도, 말씀읽기 부족 일수에 따른 벌금 계산을 수동으로 해야 한다.

토요 목자모임 지각 벌금 계산이 번거롭다.

납부 여부를 별도로 확인하고 관리해야 한다.

구성원들이 입력이나 투표를 놓쳤을 때 반복적으로 안내해야 한다.

관리자는 납부 요청이 들어왔는지 별도로 확인해야 한다.

캠퍼스별 운영 기준이 다를 수 있어 향후 확장 가능한 구조가 필요하다.

## 일반 구성원

일반 구성원은 경건생활을 입력하고, 투표에 응답하며, 자신의 벌금과 납부 상태를 확인하는 사용자이다.

일반 구성원은 앱에서 주간 큐티 일수, 기도 일수, 말씀읽기 일수를 입력할 수 있다.

토요 목자모임에서 지각한 경우 지각 시간을 입력할 수 있다.

수요예배 참여 투표와 토요 목자모임 참여 투표에 응답할 수 있다.

점심 투표에 응답할 수 있다.

자신의 벌금 내역을 확인하고, 납부 후 납부 요청을 보낼 수 있다.

앱 푸시 알림을 통해 경건생활 입력 요청, 투표 안내, 정산 결과, 납부 안내를 받을 수 있다.

## 관리자

관리자는 구성원들의 입력 상태와 투표 상태, 정산 결과, 납부 상태를 관리하는 사용자이다.

관리자는 앱 내 관리자 화면에서 구성원을 관리할 수 있다.

경건생활 미입력자를 확인할 수 있다.

수요예배와 토요 목자모임 참여 투표의 미응답자를 확인할 수 있다.

점심 투표를 생성하고 결과를 확인할 수 있다.

반복 투표를 설정할 수 있다.

벌금 규칙을 설정할 수 있다.

정산을 생성하고 개인별 벌금 내역을 확인할 수 있다.

사용자의 납부 요청을 승인하거나 반려할 수 있다.

미납자를 확인할 수 있다.

필요한 대상에게 푸시 알림을 발송할 수 있다.

## 제외 기능

카카오톡 메시지 자동 수집은 제공하지 않는다.

카카오톡 공유 문구 생성은 제공하지 않는다.

결제 API 연동은 제공하지 않는다.

QR 체크인은 초기 범위에서 제외한다.

GPS 출석 인증은 초기 범위에서 제외한다.

점심 밥 투표, 정상 기능은 초기 범위에서 제외한다.

AI 묵상 분석은 제공하지 않는다.

앱 내 묵상 본문 저장은 제공하지 않는다.

복잡한 통계 기능은 초기 범위에서 제외한다.

포인트, 랭킹, 캐릭터 성장 기능은 제공하지 않는다.

Supabase 클라이언트를 앱에서 직접 사용하지 않는다.

## 주요 기능

## 회원가입 및 로그인

사용자는 이메일, 비밀번호, 이름을 입력하여 회원가입할 수 있다.

비밀번호는 암호화하여 저장한다.

로그인에 성공하면 JWT를 발급한다.

사용자는 로그인 후 자신의 정보와 소속 그룹 정보를 확인할 수 있다.

관리자와 일반 구성원의 권한을 구분하여 접근을 제어한다.

앱은 로그인 후 사용자의 FCM 토큰을 서버에 등록한다.

로그아웃 시 서버에 저장된 FCM 토큰을 비활성화한다.

---

## 캠퍼스 및 그룹 관리

서비스는 캠퍼스와 그룹 구조를 가진다.

하나의 캠퍼스는 여러 그룹을 가질 수 있고, 하나의 그룹은 여러 구성원을 가질 수 있다.

초기에는 우리 캠퍼스 기준으로만 운영한다.

사용자는 관리자가 제공한 초대코드를 입력하여 그룹에 가입한다.

그룹 내 역할은 소유자, 관리자, 일반 구성원으로 구분한다.

관리자는 구성원의 상태와 역할을 관리할 수 있다.

---

## 경건생활 일수 입력

경건생활은 매일 체크하는 방식이 아니라 주간 일수 입력 방식으로 관리한다.

사용자는 매주 일요일에 한 주 동안 큐티, 기도, 말씀읽기를 각각 며칠 수행했는지 입력한다.

기본 기준은 한 주에 다섯 날이다.

큐티는 묵상 본문을 앱에 저장하지 않고, 사용자가 수행 일수를 직접 입력하는 방식으로 운영한다.

사용자가 입력을 완료하면 시스템은 벌금 미리보기를 제공한다.

입력 완료 시 제출 시간이 저장된다.

관리자는 경건생활을 입력하지 않은 구성원을 확인할 수 있다.

월요일 자동 점검 시 제출 시간이 없는 사용자는 경건생활 미입력자로 판단한다.

---

## 경건생활 벌금 계산

경건생활 벌금은 기준 일수에서 부족한 일수를 기준으로 계산한다.

큐티와 기도는 부족한 하루마다 오백 원으로 계산한다.

말씀읽기는 부족한 하루마다 삼백 원으로 계산한다.

시스템은 각 항목별 벌금을 계산한 뒤 경건생활 벌금 합계를 보여준다.

---

## 토요 목자모임 지각 기록

토요 목자모임은 지각 시간을 벌금 계산에 반영한다.

사용자는 지각 여부와 지각 시간을 입력할 수 있다.

지각 벌금은 기본 천 원에 지각한 분마다 백 원을 더해 계산한다.

불참 벌금은 초기 범위에서는 적용하지 않는다.

필요한 경우 불참은 기록용 상태로만 관리할 수 있다.

---

## 반복 참여 투표

수요예배 참여 투표와 토요 목자모임 참여 투표는 매주 동일한 형식으로 진행되므로 반복 투표로 제공한다.

관리자는 반복 투표 템플릿을 한 번 설정하고, 시스템은 매주 해당 설정에 따라 투표를 자동 생성한다.

수요예배 참여 투표는 참여, 미참여, 미정 등의 선택지를 가질 수 있다.

토요 목자모임 참여 투표는 정시 참석, 지각 예정, 미정 등의 선택지를 가질 수 있다.

사용자는 홈 화면에서 현재 진행 중인 반복 투표를 확인하고 응답할 수 있다.

관리자는 투표 결과와 미응답자를 확인할 수 있다.

시스템은 반복 투표가 시작될 때 대상 구성원에게 푸시 알림을 발송할 수 있다.

시스템은 마감 시간이 가까워졌을 때 미응답자에게 푸시 알림을 발송할 수 있다.

---

## 점심 자유 투표(MVP제외)

점심 투표는 매주 식당, 메뉴, 밥 양 등 형식이 달라질 수 있으므로 자유 투표 방식으로 제공한다.

관리자는 투표 제목과 선택지를 직접 입력할 수 있다.

예를 들어 식당 투표를 만들 수도 있고, 특정 식당이 정해진 후 메뉴 투표나 밥 양 투표를 만들 수도 있다.

점심 투표는 기본적으로 비익명, 단일 선택 방식으로 제공한다.

관리자는 점심 투표 결과와 미응답자를 확인할 수 있다.

점심 투표가 생성되면 시스템은 구성원에게 푸시 알림을 발송할 수 있다.

마감 시간이 가까워졌을 때 미응답자에게 푸시 알림을 발송할 수 있다.

---

## 벌금 정산

정산은 사용자가 입력한 경건생활 일수와 지각 시간을 기준으로 자동 계산된다.

관리자는 정산 기간을 선택하여 정산을 생성한다.

시스템은 해당 기간의 경건생활 기록과 지각 기록을 조회한다.

큐티, 기도, 말씀읽기의 부족 일수를 계산하고, 지각 시간이 있는 경우 지각 벌금을 계산한다.

개인별 벌금 합계와 상세 사유를 저장한다.

사용자는 자신의 정산 내역을 확인할 수 있다.

관리자는 전체 구성원의 정산 내역을 확인할 수 있다.

정산이 생성되면 시스템은 대상 구성원에게 정산 결과 알림을 발송할 수 있다.

월요일 자동 점검 시 총 벌금이 있고 납부 완료 상태가 아닌 사용자는 미납 알림 대상이 된다.

단, 사용자가 이미 납부 요청을 보낸 경우에는 사용자에게 미납 알림을 보내지 않고 관리자에게 납부 확인 요청 알림을 보낸다.

---

## 납부 관리

서비스는 결제 기능을 제공하지 않는다.

벌금은 기존 방식처럼 계좌이체로 납부하고, 서비스는 납부 상태만 관리한다.

사용자는 자신의 벌금 금액을 확인한 뒤 계좌이체로 납부한다.

납부 후 사용자는 서비스에서 납부 요청을 보낸다.

관리자는 실제 입금 내역을 확인한 뒤 납부 완료 또는 반려로 처리한다.

납부 상태는 미납, 확인 요청, 납부 완료, 반려, 면제, 이월로 관리한다.

사용자가 납부 요청을 보내면 관리자는 푸시 알림 또는 관리자 화면에서 확인 요청을 확인할 수 있다.

관리자가 납부 완료 또는 반려 처리를 하면 사용자에게 처리 결과 알림을 발송할 수 있다.

---

## 월요일 자동 점검

시스템은 매주 월요일에 자동 점검을 수행한다.

자동 점검은 경건생활 입력 여부와 벌금 납부 상태를 확인한다.

경건생활 기록의 제출 시간이 없는 사용자는 미입력자로 판단한다.

경건생활 미입력자에게는 입력 요청 알림을 발송한다.

정산 결과 기준으로 총 벌금이 있고 납부 완료 상태가 아닌 사용자는 납부 상태를 확인한다.

납부 상태가 미납 또는 반려인 경우 사용자에게 납부 요청 알림을 발송한다.

납부 상태가 확인 요청인 경우 사용자에게 미납 알림을 보내지 않고 관리자에게 납부 확인 요청 알림을 발송한다.

납부 상태가 납부 완료, 면제, 이월인 경우 사용자에게 미납 알림을 보내지 않는다.

중복 알림을 방지하기 위해 같은 주차, 같은 사용자, 같은 알림 유형에 대해 발송 이력을 저장한다.

---

## FCM 푸시 알림

앱은 사용자 기기의 FCM 토큰을 발급받아 서버에 등록한다.

서버는 사용자별 FCM 토큰을 저장하고, 필요한 상황에 대상 사용자에게 푸시 알림을 발송한다.

한 사용자가 여러 기기를 사용할 수 있으므로 FCM 토큰은 사용자와 별도 테이블로 관리한다.

사용자가 로그아웃하거나 알림 수신을 해제한 경우 해당 토큰을 비활성화할 수 있다.

푸시 알림 전송 실패가 발생한 경우 서버는 실패 토큰을 비활성화하거나 재등록을 유도한다.

알림 발송 이력을 저장하여 중복 발송을 방지하고, 실패 원인을 추적할 수 있도록 한다.

## 데이터베이스 설계 방향

데이터베이스는 Supabase PostgreSQL을 사용한다.

Spring Boot 서버는 PostgreSQL JDBC Driver를 통해 Supabase DB에 연결한다.

앱은 Supabase DB에 직접 접근하지 않는다.

DB 접속 정보와 비밀번호는 코드에 직접 작성하지 않고 환경변수로 관리한다.

개발 초기에는 JPA의 자동 테이블 생성 기능을 사용할 수 있지만, 운영 단계에서는 validate 설정 또는 마이그레이션 도구를 사용하는 방향으로 전환한다.

Supabase 무료 플랜의 연결 수 제한을 고려하여 HikariCP 커넥션 풀 크기를 작게 설정한다.

알림 발송 이력과 사용자 FCM 토큰도 Supabase PostgreSQL에 저장한다.

관리자는 납부 요청을 승인하거나 반려할 수 있다.

면제나 이월 처리가 필요한 경우 관리자 권한으로 처리할 수 있도록 한다.

납부 요청, 납부 승인, 납부 반려 시 푸시 알림을 발송할 수 있도록 한다.

---

## 관리자 대시보드

관리자가 운영에 필요한 정보를 한눈에 볼 수 있도록 대시보드를 구성한다.

대시보드에서는 미입력자, 미응답자, 미납자, 납부 확인 요청 목록을 확인할 수 있다.

관리자는 대시보드에서 필요한 대상에게 알림을 발송할 수 있다.

---

## 배포와 운영

서비스를 배포하여 실제 캠퍼스 구성원을 대상으로 운영한다.

운영 중 발생하는 피드백을 바탕으로 입력 방식, 투표 방식, 정산 방식, 관리자 화면, 알림 정책을 개선한다.

---

## 이력서 작성 방향

교회 캠퍼스의 경건생활 입력, 예배 및 모임 참여 투표, 점심 투표, 벌금 정산, 알림을 통합 관리하는 모바일 앱 서비스를 기획하고 개발했다.

Spring Boot 기반 REST API와 React Native 앱을 분리하여 개발했고, Supabase PostgreSQL을 데이터베이스로 사용했다.

주간 경건생활 일수와 토요 목자모임 지각 시간을 기반으로 개인별 벌금을 자동 계산했다.

JWT 기반 인증과 역할 기반 권한 관리를 적용했고, 캠퍼스와 그룹 단위의 데이터 구조를 설계했다.

Spring Scheduler를 활용하여 매주 반복되는 참여 투표를 자동 생성하고, 일요일 경건생활 입력 안내와 월요일 미입력자 및 미납자 점검 알림을 자동 발송하도록 구현했다.

Firebase Cloud Messaging을 연동하여 경건생활 입력 요청, 투표 미응답, 정산 결과, 납부 상태 변경 알림을 발송하도록 구현했다.

실제 공동체에서 사용하는 운영 서비스를 목표로 개발하여, 수기 운영 방식의 문제를 데이터 기반 관리 방식으로 개선했다.

## 결론

FaithLog는 교회 캠퍼스의 반복적인 운영 업무를 줄이고, 구성원들이 경건생활 입력, 예배 및 모임 참여 투표, 점심 투표, 벌금 납부 상태를 쉽게 관리할 수 있도록 돕는 모바일 앱 서비스이다.

초기에는 우리 캠퍼스 기준으로 실제 운영 가능한 MVP를 개발하고, 향후 여러 캠퍼스에서도 사용할 수 있도록 캠퍼스와 그룹 중심의 확장 가능한 구조를 갖춘다.

핵심 목표는 많은 기능을 넣는 것이 아니라, 실제 운영에 필요한 경건생활 입력, 반복 투표, 자유 투표, 벌금 정산, 납부 상태 관리, FCM 알림, Supabase 기반 데이터 관리를 안정적으로 구현하는 것이다.
