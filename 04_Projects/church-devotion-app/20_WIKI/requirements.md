## 기능 요구사항

### 인증/사용자

- 사용자는 이메일, 비밀번호, 이름으로 회원가입할 수 있어야 한다.
- 로그인 성공 시 Access Token과 Refresh Token을 HttpOnly Cookie로 내려줘야 한다.
- Refresh Token Cookie의 Path는 `/api/v1/auth/refresh`로 제한한다.
- 로그아웃 시 Access Token과 Refresh Token Cookie를 만료시킨다.
- 사용자는 `GET /users/me`로 본인 정보와 캠퍼스 멤버십을 조회할 수 있어야 한다.

### 캠퍼스/멤버

- `MANAGER`, `ADMIN`은 캠퍼스를 생성할 수 있어야 한다.
- 캠퍼스 생성자는 해당 캠퍼스의 `CAMPUS_LEADER`로 등록한다.
- 사용자는 초대코드로 캠퍼스에 가입하며, 가입 승인 없이 `ACTIVE` 상태가 된다.
- 캠퍼스 내부 역할 변경과 멤버 비활성화는 `ADMIN`, `MINISTER`, `ELDER`, `CAMPUS_LEADER`가 할 수 있어야 한다.
- 한 사용자는 여러 캠퍼스에 소속될 수 있어야 한다.

### 경건생활

- 사용자는 날짜별로 큐티, 기도, 말씀읽기 체크를 저장할 수 있어야 한다.
- 하루 체크는 `submitted_at`을 변경하지 않고, `charge_items`를 생성하지 않는다.
- 사용자는 주간 단위로 일괄 저장 및 제출할 수 있어야 한다.
- 주간 제출 시 월요일부터 일요일까지 7일치 체크 row를 보장한다.
- 요청에 없는 날짜는 false 기본값으로 저장한다.
- `submit = true`이면 `submitted_at`을 갱신하고 벌금 청구를 생성 또는 갱신한다.
- 관리자는 주차별 제출 현황과 미제출자를 조회할 수 있어야 한다.

### 벌금 규칙/청구

- 벌금 규칙은 캠퍼스별로 관리한다.
- 큐티, 기도는 부족 1회당 500원, 말씀읽기는 부족 1회당 300원이 기본값이다.
- 토요 목자모임 지각은 기본 1,000원 + 지각 1분당 100원이 기본값이다.
- 한 주의 벌금은 항목별로 따로 청구하지 않고 `PENALTY` 청구 1줄로 합산한다.
- 중복 방지 기준은 `campus_id + user_id + payment_category + source_type + source_id`다.
- 사용자는 본인의 `UNPAID` 청구 항목만 `PAID`로 변경할 수 있어야 한다.
- 관리자는 캠퍼스 청구 목록과 회원별 청구 상세를 조회할 수 있어야 한다.
- 관리자는 청구 상태를 `UNPAID`, `PAID`, `WAIVED`, `CANCELED` 중 하나로 변경할 수 있어야 한다.

### 납부 계좌

- 납부 계좌는 캠퍼스와 `PENALTY`/`COFFEE` account type 기준으로 관리한다.
- 현재 계좌는 `is_active = true`로 판단한다.
- 계좌가 변경되어도 과거 청구 내역이 바뀌지 않도록 청구 생성 시점의 은행명, 계좌번호, 예금주 snapshot을 저장한다.

### 투표/커피

- 수요예배, 토요 목자모임, 커피, 커스텀 투표를 같은 `polls` 구조로 관리한다.
- 반복 투표 템플릿은 수요예배와 토요 목자모임처럼 매주 반복되는 투표에 사용한다.
- 반복 투표 템플릿의 기본 선택지는 `poll_template_options`에 저장한다.
- 사용자는 진행 중인 투표 목록과 투표 상세를 조회하고 응답할 수 있어야 한다.
- 한 사용자는 한 투표에 하나의 응답만 가질 수 있다.
- 투표는 `SINGLE` 또는 `MULTIPLE` 선택 방식을 지원한다.
- 투표 상태는 `SCHEDULED`, `OPEN`, `CLOSED`로 관리한다.
- 커피 투표는 `chargeGenerationType = OPTION_PRICE`, `paymentCategory = COFFEE`를 사용한다.
- 커피 투표는 연결된 `paymentAccountId`를 통해 커피 계좌를 지정할 수 있어야 한다.
- 커피 투표 응답 시 선택한 옵션 금액이 0보다 크면 `COFFEE` 청구를 생성 또는 갱신한다.
- `안 먹어요`처럼 금액이 0인 선택지는 청구 항목을 생성하지 않는다.
- 커피 선택지는 `compose_menu_code`와 `price_amount`를 저장해 메뉴 코드와 가격 snapshot을 분리한다.
- 관리자는 투표 결과와 미참여자 목록을 조회할 수 있어야 한다.
- 투표 상세에는 댓글과 대댓글을 남길 수 있어야 하며, 삭제는 soft delete로 처리한다.

### 커피 담당

- 커피 담당은 캠퍼스 내부 역할에 넣지 않고 `campus_duty_assignments`로 분리한다.
- 현재 커피 담당자는 `duty_type = COFFEE`, `is_active = true`인 사용자다.
- MVP에서는 캠퍼스당 활성 커피 담당자 1명을 권장한다.
- 커피 담당자는 커피 투표 생성, 커피 계좌 관리, 커피 주문 확인을 할 수 있어야 한다.

### 알림/대시보드

- 앱은 사용자별 FCM 토큰을 서버에 등록하고 비활성화할 수 있어야 한다.
- 서버는 경건생활 미제출자, 투표 미참여자, 커스텀 대상에게 FCM 알림을 발송할 수 있어야 한다.
- 알림 유형은 경건 리마인드/미제출, 수요예배 투표 오픈/미응답, 토요 목자모임 투표 오픈/미응답, 커피 투표 오픈/미응답, 미납, 커스텀으로 구분한다.
- 알림 로그는 유형, 대상 주차, 대상 ID, 발송 상태, 실패 사유를 저장해야 한다.
- 대시보드는 멤버 수, 경건생활 제출률, 청구 요약, 투표 진행 현황을 제공해야 한다.

## 비기능 요구사항

- 앱은 Supabase DB에 직접 접근하지 않는다. 모든 요청은 Spring Boot API 서버를 통한다.
- DB 비밀번호와 연결 정보는 환경변수로 관리한다.
- Supabase 무료 플랜의 연결 수 제한을 고려해 HikariCP pool size를 작게 둔다.
- 개발 초기에 JPA `ddl-auto update`를 사용할 수 있지만 운영 전에는 validate 또는 마이그레이션 방식으로 전환한다.
- Refresh Token은 원본이 아니라 hash만 DB에 저장한다.
- API 응답은 공통 성공/실패 포맷을 사용한다.

## 관리자 요구사항

- 전역 `ADMIN`은 사용자 전체 권한을 변경할 수 있다.
- 캠퍼스 운영 권한자는 캠퍼스 정보, 멤버 역할, 커피 담당, 벌금 규칙, 투표, 계좌, 청구 상태, 알림을 관리할 수 있다.
- 관리자는 미제출자, 미응답자, 미납자를 각각 조회할 수 있어야 한다.
- 관리자는 필요 대상에게 수동 알림을 보낼 수 있어야 한다.

## 알림/인증/권한 요구사항

- 서비스 전체 권한: `USER`, `MANAGER`, `ADMIN`
- 캠퍼스 내부 역할: `MINISTER`, `ELDER`, `CAMPUS_LEADER`, `MEMBER`
- 운영 담당: `COFFEE`
- 멤버 상태: `ACTIVE`, `INACTIVE`
- 청구 상태: `UNPAID`, `PAID`, `WAIVED`, `CANCELED`
- 미제출자 기준: 활성 캠퍼스 멤버 중 해당 주차 `weekly_devotion_records`가 없거나 `submitted_at`이 null인 사용자
- 미납자 기준: `charge_items.status = UNPAID`

## 보류된 요구사항

- 점심 관련 전체 기능
- 관리자 납부 승인/반려 전용 플로우
- 납부 요청 상태, 반려 상태, 이월 상태
- 입금 인증 사진
- 결제 API
- 카카오톡 자동 연동
- QR/GPS 기반 출석 인증
- 복잡한 통계, 포인트/랭킹/캐릭터 성장
