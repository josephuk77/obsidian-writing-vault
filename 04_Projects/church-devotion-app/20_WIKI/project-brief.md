## 프로젝트 한 줄 설명

FaithLog는 교회 캠퍼스의 경건생활 체크, 예배/모임 참여 투표, 커피 주문, 벌금/커피 청구, 납부 상태, 푸시 알림을 한 곳에서 관리하는 모바일 앱 서비스이다.

## 문제 정의

현재 캠퍼스 운영은 경건생활 입력 여부, 수요예배 참여 여부, 토요 목자모임 참여 여부, 커피 주문, 벌금 계산, 납부 확인, 미응답자 안내가 분리되어 있다. 관리자는 매주 같은 확인 작업을 반복하고, 구성원은 입력/응답/납부를 놓치기 쉽다.

FaithLog는 반복 운영 업무를 데이터로 모으고, 누락자를 자동으로 찾고, 필요한 사람에게 알림을 보내는 운영 도구다. 핵심은 묵상 콘텐츠 서비스가 아니라 캠퍼스 운영 흐름을 줄이는 것이다.

## 대상 사용자

| 사용자 | 하는 일 | 필요한 가치 |
| --- | --- | --- |
| 일반 구성원 | 경건생활 체크, 투표 응답, 커피 주문, 청구 확인, 납부 완료 처리 | 매주 해야 할 일을 앱에서 빠르게 끝낸다 |
| 캠퍼스 운영자 | 구성원 관리, 투표 생성, 미응답자 확인, 청구/납부 현황 확인, 알림 발송 | 수기 확인과 반복 안내를 줄인다 |
| 커피 담당자 | 커피 투표 생성, 주문 확인, 커피 계좌 관리 | 커피 주문과 커피비 청구를 한 흐름으로 처리한다 |
| 서비스 관리자 | 사용자 전체 권한과 캠퍼스 생성 권한 관리 | 여러 캠퍼스 확장 시 전역 권한을 통제한다 |

## 핵심 가치

- 구성원은 이번 주 해야 할 입력, 투표, 납부 상태를 한 화면에서 확인한다.
- 관리자는 미제출자, 미응답자, 미납자를 직접 대조하지 않는다.
- 벌금과 커피비는 `charge_items` 하나의 청구 흐름으로 관리한다.
- 알림 발송 이력을 남겨 같은 주차, 같은 대상, 같은 유형의 중복 알림을 막는다.
- 초기에는 한 캠퍼스에서 실제 운영 가능한 형태로 만들되, 데이터 구조는 여러 캠퍼스를 감당한다.

## MVP 범위

- 회원가입, 로그인, 쿠키 기반 JWT 인증, 로그아웃
- 캠퍼스 생성, 초대코드 기반 가입, 캠퍼스 멤버 관리
- 서비스 전체 권한과 캠퍼스 내부 역할 분리
- 커피 담당자 지정 및 해제
- 하루별 경건생활 체크와 주간 제출
- 주간 제출 시 벌금 자동 계산 및 `PENALTY` 청구 생성
- 수요예배/토요 목자모임 반복 투표 템플릿과 실제 투표
- 투표 댓글과 대댓글
- 커피 주문 투표와 `COFFEE` 청구 생성
- 내 청구 항목/요약 조회, 사용자 직접 납부 완료 처리
- 관리자 청구 조회, 관리자 청구 상태 변경
- FCM 토큰 등록/비활성화, 미제출/미응답/커스텀 알림, 알림 로그
- 관리자 대시보드

## 제외 범위

- 점심 투표, 점심 공동 주문, 점심 금액 배분, 점심 담당자, 점심 계좌
- 카카오톡 메시지 자동 수집 및 카카오톡 공유 문구 생성
- 결제 API 연동
- QR 체크인, GPS 출석 인증
- AI 묵상 분석, 앱 내 묵상 본문 저장
- 포인트, 랭킹, 캐릭터 성장
- Supabase 클라이언트 직접 사용
- 관리자 납부 승인 플로우는 현재 MVP 기준에서 제외한다. 최신 ERD/API는 사용자 직접 `PAID` 처리를 기준으로 한다.

## 성공 기준

- 구성원이 1분 안에 이번 주 경건생활 제출과 진행 중 투표 응답을 끝낼 수 있다.
- 관리자가 특정 주차의 미제출자, 미응답자, 미납자를 한 화면에서 확인할 수 있다.
- 주간 경건생활 제출 후 벌금 청구가 중복 없이 생성 또는 갱신된다.
- 커피 투표 응답 후 커피 청구가 중복 없이 생성 또는 갱신된다.
- 월요일 점검과 수동 알림이 중복 발송 이력을 남긴다.

## 최신 기준

최신 기준은 `[[ERD설계]]`와 `[[API 명세서]]`다. `[[기획서]]`에는 이전 구상인 그룹 구조, 점심 투표, 관리자 납부 승인 흐름이 섞여 있으므로 구현 기준으로 바로 사용하지 않는다.

2026-06-12 DB 수정 기준으로는 `poll_template_options`, `poll_response_options`, `poll_comments`, `payment_accounts`, `notification_logs.notification_type`까지 MVP 구현 기준에 포함한다. 밥/식사/점심 기능은 계속 MVP 제외다.
